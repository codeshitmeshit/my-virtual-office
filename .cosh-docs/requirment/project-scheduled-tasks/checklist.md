# 项目定时任务 Checklist

确认状态：已确认

## Phase 1：扩展现有 Cron 支持项目绑定

- [ ] CHK-001 现有 cron job 支持保存项目绑定元数据。
  - 验证方法：通过现有 cron 创建/更新能力保存带 `projectId`、`targetType`、可选 `taskId` 的项目定时任务。
  - 预期结果：合法项目定时任务能被创建、查询、更新、启用/禁用和删除；底层仍表现为 cron job。
  - 关联需求点：复用现有 Cron 作为底层调度器；项目定时任务必须绑定项目。

- [ ] CHK-002 项目绑定元数据在服务重启或 cron 存储重载后仍保留。
  - 验证方法：创建多个带项目绑定的 cron job 后重载存储或重启服务，再查询 cron job。
  - 预期结果：`projectId`、调度规则、目标类型、启用状态和项目元数据均正确保留。
  - 关联需求点：项目定时任务复用 cron 存储且必须持久化。

- [ ] CHK-003 不满足条件的项目不能创建项目绑定 cron。
  - 验证方法：对没有负责人且没有绑定 Agent 的项目创建带 `projectId` 的 cron job。
  - 预期结果：请求被拒绝，并返回清晰错误；不会保存半成品记录。
  - 关联需求点：只有有负责人或绑定 Agent 的项目才能配置。

- [ ] CHK-004 复用现有 Cron 调度规则校验，覆盖 cron、循环间隔和一次性时间。
  - 验证方法：分别提交三类合法和非法 schedule。
  - 预期结果：项目绑定 cron 与普通 Agent cron 使用一致的 schedule 校验；合法规则被接受，非法规则被拒绝并给出可理解错误。
  - 关联需求点：支持固定时间、循环间隔和一次性执行。

- [ ] CHK-005 项目目标元数据校验覆盖整个项目和指定任务。
  - 验证方法：创建目标为整个项目、当前项目已有任务、不存在任务、其他项目任务的项目绑定 cron。
  - 预期结果：整个项目和当前项目任务被接受；非法任务目标被拒绝。
  - 关联需求点：支持项目 workflow 和指定项目任务。

- [ ] CHK-006 普通 Agent 级 Cron Manager 行为不受项目绑定扩展影响。
  - 验证方法：使用 `/cron.html` 创建、查询、运行 Agent 级 cron，或运行现有 cron 相关测试。
  - 预期结果：不带 `projectId` 的 Agent 级 cron 仍然可用，不要求填写项目字段，原 payload 语义保持兼容。
  - 关联需求点：兼容现有 Cron Manager。

## Phase 2：项目详情页配置体验与全局定时任务总览

- [ ] CHK-007 项目详情页只对满足条件的项目显示定时任务配置能力。
  - 验证方法：分别打开满足条件和不满足条件的项目。
  - 预期结果：满足条件的项目可配置；不满足条件的项目不提供创建入口或展示明确原因。
  - 关联需求点：按负责人或绑定 Agent 限制使用边界。

- [ ] CHK-008 项目负责人可以在项目详情页创建整个项目 workflow 定时任务。
  - 验证方法：通过项目 UI 创建目标为整个项目 workflow 的定时任务。
  - 预期结果：定时任务出现在项目定时任务列表中，并显示正确的时间规则、目标、启用状态和下次运行时间。
  - 关联需求点：项目负责人可配置项目 workflow 定时启动。

- [ ] CHK-009 项目负责人可以创建重复执行指定任务的定时任务。
  - 验证方法：通过项目 UI 选择某个项目任务作为定时目标。
  - 预期结果：只能选择当前项目内任务；保存后能清楚看到目标任务。
  - 关联需求点：指定任务可作为固定周期动作重复执行。

- [ ] CHK-010 项目详情页支持编辑、启用/禁用、删除和立即运行。
  - 验证方法：在项目页逐一操作这些控制项。
  - 预期结果：UI 状态和后端数据保持一致。
  - 关联需求点：项目负责人可管理定时任务生命周期。

- [ ] CHK-011 项目级暂停/恢复影响该项目全部定时任务。
  - 验证方法：暂停项目定时任务，查看任务列表状态，再恢复并再次查看。
  - 预期结果：暂停状态清晰可见；单个任务配置仍保留；恢复后允许继续运行。
  - 关联需求点：项目级暂停/恢复总开关。

- [ ] CHK-012 UI 文案保持项目语境，不泛化成自动化平台。
  - 验证方法：检查中英文标签、空状态、弹窗和提示语。
  - 预期结果：用户感知为“项目定时任务”，不是办公室级自动化规则。
  - 关联需求点：产品概念和范围边界。

- [ ] CHK-031 全局定时任务页可以看到项目绑定 cron。
  - 验证方法：创建普通 Agent cron 和项目绑定 cron 后打开 `/cron.html`。
  - 预期结果：全局定时任务页同时展示普通 Agent cron 和项目绑定 cron，并能区分类型。
  - 关联需求点：全局可见性；用户能知道系统当前有哪些定时任务。

- [ ] CHK-032 全局定时任务页展示项目上下文。
  - 验证方法：在 `/cron.html` 查看项目绑定 cron 的列表项或详情。
  - 预期结果：项目绑定 cron 显示项目名、目标类型、目标任务名或“整个项目 workflow”，并可跳转或定位到对应项目。
  - 关联需求点：系统级排查和项目绑定语义。

- [ ] CHK-033 全局定时任务页支持按类型或项目过滤。
  - 验证方法：在 `/cron.html` 使用过滤控件筛选普通 Agent cron、项目 cron、指定项目。
  - 预期结果：过滤结果准确，不影响原有 Agent 过滤能力。
  - 关联需求点：全局定时任务管理可用性。

## Phase 3：调度派发与项目执行绑定

- [ ] CHK-013 到期的整个项目定时任务在项目空闲时启动 workflow。
  - 验证方法：为一个空闲项目创建到期的项目 workflow 定时任务，并触发调度处理。
  - 预期结果：项目 workflow 启动，定时任务运行状态被更新。
  - 关联需求点：长期项目自动唤醒和继续推进。

- [ ] CHK-014 到期的整个项目定时任务在 workflow 已运行时跳过。
  - 验证方法：先启动项目 workflow，再触发到期的整个项目定时任务。
  - 预期结果：不会重复启动 workflow；本次运行记录为 skipped，并说明 workflow 已运行。
  - 关联需求点：workflow 冲突处理。

- [ ] CHK-015 到期的指定任务定时任务启动对应项目任务。
  - 验证方法：创建到期的指定任务定时任务并触发调度处理。
  - 预期结果：被选中的任务进入预期执行流程，运行记录关联该任务。
  - 关联需求点：指定项目任务可周期执行。

- [ ] CHK-016 项目级暂停会阻止调度派发。
  - 验证方法：暂停项目定时任务后，让某个定时任务到期并触发调度处理。
  - 预期结果：不会启动 workflow，也不会启动任务执行。
  - 关联需求点：项目级暂停/恢复控制。

- [ ] CHK-017 已删除、归档、缺失或不再满足条件的目标不会被派发。
  - 验证方法：构造已删除项目、归档项目、已删除任务或不再满足条件的项目定时任务。
  - 预期结果：调度器拒绝派发并记录或报告原因，不崩溃。
  - 关联需求点：安全执行边界和兼容性。

- [ ] CHK-018 调度决策后正确更新下次运行和上次运行字段。
  - 验证方法：分别触发 cron、循环间隔、一次性、跳过和失败场景。
  - 预期结果：`lastRunAt`、`lastStatus`、`nextRunAt` 与实际结果一致。
  - 关联需求点：可见调度状态和可靠循环执行。

## Phase 4：执行历史与控制面板异常提示

- [ ] CHK-019 项目详情页展示定时执行历史。
  - 验证方法：触发 started、skipped、failed、completed、intervention_required 等状态后打开项目详情。
  - 预期结果：历史记录展示时间、定时任务名称、目标、状态和原因/错误信息。
  - 关联需求点：项目级执行历史。

- [ ] CHK-020 正常定时执行不会制造控制面板噪音。
  - 验证方法：触发成功启动/完成，以及 workflow 已运行导致的正常跳过。
  - 预期结果：项目历史记录这些运行；控制面板不显示普通运行提示。
  - 关联需求点：控制面板只提示异常。

- [ ] CHK-021 失败状态会同时出现在项目页和控制面板。
  - 验证方法：强制制造一次调度派发失败。
  - 预期结果：项目页标记失败，控制面板显示可处理的失败提示。
  - 关联需求点：失败处理和负责人可见性。

- [ ] CHK-022 需要人工介入状态会同时出现在项目页和控制面板。
  - 验证方法：模拟一次需要人工介入的运行状态。
  - 预期结果：项目页和控制面板都显示人工介入提示，并提供足够上下文。
  - 关联需求点：人工介入处理。

- [ ] CHK-023 执行历史在长期使用下保持可读。
  - 验证方法：制造超过展示上限的多次定时运行记录。
  - 预期结果：UI 仍响应正常，历史按时间排序，并通过过滤、分页或截断保持可读。
  - 关联需求点：长期项目可用性。

- [ ] CHK-024 调度诊断信息足够排查问题。
  - 验证方法：查看 due、skipped、failed、paused 等调度决策后的服务端日志或调试输出。
  - 预期结果：日志包含项目、定时任务、目标类型、决策和错误原因，不暴露无关数据。
  - 关联需求点：可观测性。

## Phase 5：长期项目推荐、体验打磨和回归加固

- [ ] CHK-025 长期项目设置流程推荐创建定时任务，但不自动启用。
  - 验证方法：创建或标记一个长期项目，并完成项目设置流程。
  - 预期结果：UI 推荐添加定时任务；除非用户确认，否则不会启用任何定时任务。
  - 关联需求点：长期项目默认策略。

- [ ] CHK-026 没有定时任务数据的旧项目可以正常加载。
  - 验证方法：打开没有任何定时任务字段的既有项目。
  - 预期结果：项目详情无报错；定时任务区域能显示空状态。
  - 关联需求点：向后兼容。

- [ ] CHK-027 新增 UI 的本地化覆盖完整。
  - 验证方法：运行 locale 完整性检查，并人工检查中英文 UI。
  - 预期结果：新增标签、空状态、校验错误、状态和控制面板提示都有中英文文案。
  - 关联需求点：中英文 UI 支持。

- [ ] CHK-028 回归测试覆盖现有项目执行和 workflow 路径。
  - 验证方法：运行现有项目 workflow/项目执行测试以及新增定时任务测试。
  - 预期结果：旧行为保持通过，新行为有测试覆盖。
  - 关联需求点：不破坏现有项目系统。

- [ ] CHK-029 回归测试覆盖现有 Agent 级 Cron Manager。
  - 验证方法：运行现有 cron 相关测试，并手动 smoke-test `/cron.html`。
  - 预期结果：Agent 级 cron 仍保持独立可用。
  - 关联需求点：兼容现有 Cron Manager。

- [ ] CHK-030 端到端人工验收覆盖 5 个 phase 的完整链路。
  - 验证方法：创建满足条件的长期项目，接受推荐，创建整个项目和指定任务两类定时任务，触发到期执行，暂停/恢复项目定时任务，检查历史和控制面板提示。
  - 预期结果：项目定时任务作为一个完整项目级体验可用。
  - 关联需求点：整体产品成功标准。
