# Auto Project Workspace Review

## Review Conclusion

Reviewed with no blocking product questions. The desired default behavior is clear: new projects should be executable by default, and an omitted workspace path should no longer imply a normal project. Instead, the system should create a formal project-owned workspace automatically.

The main technical risk is data-safety around automatic workspace lifecycle. The implementation must distinguish automatically created workspaces from user-provided workspaces so deletion can ask the correct question and avoid deleting user-owned directories by mistake.

## Product Review

The product direction is coherent:

- It reduces project creation friction for users who do not know what path to enter.
- It aligns the default creation flow with the goal of increasing Project Execution usage.
- It keeps normal projects available through an explicit switch, avoiding a forced behavior change for users who only want a kanban-style project.
- It treats automatic workspaces as formal project workspaces, which matches the expectation that task execution and artifacts should work immediately after creation.
- It correctly fails project creation when automatic workspace creation fails, rather than silently creating a less capable project.

No further product clarification is required before checklist confirmation.

## Technical Review

### Project Creation State

The current frontend derives `projectExecutionEnabled` from whether the workspace input is non-empty. That must change because executable project is now a project-type choice, not merely a path-derived state.

Recommended model:

- Creation form has a project type control.
- Executable is selected by default.
- If executable and workspace path is provided, validate and bind that path.
- If executable and workspace path is empty, create an automatic workspace and bind it.
- If normal project is selected, do not create a workspace and set `projectExecutionEnabled` false.

### Workspace Provenance

The project should be able to tell whether its workspace was generated by the system. Without this, deletion cannot safely ask whether to delete the workspace, because user-provided workspace deletion must be treated differently from project-owned generated workspace deletion.

Recommended product-data addition:

- Track workspace provenance, for example `workspaceManagedBy: "system"` for automatic workspaces and `"user"` for user-provided paths.
- Optionally track `workspaceCreatedAt` or equivalent metadata for observability and future migration.

This is not a product-facing technical preference; it is needed to satisfy the confirmed deletion behavior safely.

### Automatic Workspace Naming

The name should use a sanitized project title plus timestamp. The user-facing expectation is a readable project-owned name, not a meaningless UUID-only directory.

Review points:

- Duplicate project names should not collide because timestamp is included.
- Unsafe title characters should be normalized before creating the workspace.
- Project title changes after creation do not need to rename the workspace in this phase.

### Workspace Root and Safety

The implementation must create the automatic workspace in a controlled project workspace root. It should not create random directories in arbitrary locations. The existing workspace validation and configured project-root behavior should remain the boundary for Project Execution.

If there is no usable automatic workspace root, creation should fail with a clear error.

### Delete Flow

Project deletion currently deletes project data. The new behavior adds a user decision only when the project has an automatically created workspace.

Review points:

- For system-created workspace: ask whether to delete the workspace too.
- For user-provided workspace: do not offer or perform automatic workspace deletion by default.
- If workspace deletion fails, the project deletion behavior must be clear: either block deletion and report the error, or delete project data while reporting that workspace deletion failed. This should be tested.

### Compatibility

Existing Project Execution behavior should continue:

- Existing projects with `projectExecutionEnabled` and `workspacePath` keep working.
- Existing normal projects remain normal.
- Manual workspace path creation remains supported.
- Artifact management should start working for default-created executable projects because they have a valid workspace.
- Project templates and project-from-template creation should follow the same default project type behavior where the creation UI uses the same form.

### API and Validation

The backend should not rely only on frontend defaults. API project creation must handle executable-with-empty-workspace requests directly, because users and tests can call APIs without the browser form.

Important cases:

- Executable true plus empty workspace creates automatic workspace.
- Executable true plus invalid manual workspace fails.
- Executable false plus empty workspace creates normal project.
- Executable false plus provided workspace should either ignore the path or require explicit executable selection; the product preference is to avoid accidentally enabling execution when normal project is selected.

### Security and Data Safety

Main risks:

1. Automatically creating workspaces outside the intended root.
2. Accidentally deleting user-provided workspaces.
3. Silent downgrade from executable project to normal project after creation failure.
4. Ambiguous UI where users do not realize agents may write files into the generated workspace.

Mitigations:

- Use a controlled automatic workspace root.
- Persist workspace provenance.
- Fail creation on automatic workspace failure.
- Show workspace path and status in the project UI.
- Ask before deleting system-created workspaces.

## Test Strategy

Testing should cover:

- Backend project creation for executable project without workspace.
- Backend project creation for executable project with manual workspace.
- Backend normal project creation with empty workspace.
- Frontend default project type selection.
- Frontend normal project switch.
- Generated workspace naming format.
- Workspace validation metadata after automatic creation.
- Artifact availability for a default-created executable project.
- Failure handling when automatic workspace cannot be created.
- Project deletion prompt and delete/keep outcomes for automatic workspace.
- Regression for existing Project Execution start/review/acceptance and existing project CRUD.

## Non-Blocking Follow-Ups

- Add workspace migration or "move workspace" flow.
- Add cleanup tooling for orphaned automatic workspaces.
- Add project title rename synchronization for workspace display alias.
- Add admin-level default workspace root configuration UI.
