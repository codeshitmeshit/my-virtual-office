# Meeting for AI Phase 5 方案评审

## 评审结论

**结论：通过，可进入测试 checklist 确认阶段。**

Phase 5 的产品边界已经明确：它不是完整排期系统，也不是让 Agent 自行决定是否被打断；它是在会议确认和启动阶段加入忙碌冲突检测、用户决策、advisory 推荐、轻量预约、暂停恢复和并发占用保护。

## 产品评审

### 已清晰的产品假设

- 用户最终控制权仍然保留。Advisory turn 只推荐，不执行。
- 安全性优先于启动速度。不能为了快速开会静默打断任务或重复占用 Agent。
- 轻量预约是冲突处理选项，不是完整日历功能。
- 强制加入必须有风险解释和二次确认。
- 预约时间或空闲触发不保证会议一定能启动；Agent 仍忙时必须重新进入冲突处理。

### 仍需在实现中保持的产品边界

- 不要把逻辑暂停描述为真实 provider 暂停。
- 不要把 advisory 失败解释为 Agent 同意被打断。
- 不要让预约提前阻止 Agent 当前任务。
- 不要让冲突处理流程替代普通聊天或项目执行流程。

## 技术评审

### 状态与一致性

- 会议占用校验必须是原子操作或具备等价保护，防止并发确认导致同一 Agent 被重复占用。
- 冲突记录应独立于正式会议执行状态，使用户可以等待、更换、预约或取消。
- 强制加入、等待、预约和恢复都需要持久事件记录，便于审计和重试。

### Advisory Turn

- Advisory turn 可以复用现有 Agent 通信或会议发言能力，但必须作为受控咨询事件处理。
- Advisory 输出不能直接改变会议、任务或占用状态。
- Advisory 超时、失败或无效输出必须降级为“无法获取建议”，不能默认同意强制加入。

### 任务暂停与恢复

- 需要识别 provider 能力差异：真实暂停、逻辑暂停、不可暂停。
- 原任务快照和恢复标识必须可持久化，恢复操作必须幂等。
- 恢复失败要进入可见的人工补救状态，不能隐藏在日志里。

### 轻量预约

- 预约应保存为可见安排或提醒状态，不应强锁 Agent。
- 到时重试必须重新检查冲突，而不是跳过冲突检测。
- 预约过期、取消或重新冲突的状态需要可理解文案。

### 兼容与回归

- 已完成 Phase 1-4 的用户发起会议、用户控制、AI 申请和上下文快照不能回归。
- 普通聊天、项目任务创建和项目执行不能被会议占用逻辑误拦截。
- 旧会议记录和已完成会议历史仍应正常展示。

## 主要风险

1. **占用竞争风险**：两个会议同时确认同一 Agent，导致重复占用。
   - 缓解：确认和占用写入必须具备原子保护，重复请求走幂等路径。

2. **暂停语义误导风险**：provider 不能真实暂停，但 UI 表述成真实暂停。
   - 缓解：区分真实暂停、逻辑暂停和不可暂停，并展示限制。

3. **Advisory 过度授权风险**：模型建议被当作用户确认。
   - 缓解：advisory 永远只读，所有执行动作必须由用户选择或后端明确策略触发。

4. **预约范围膨胀风险**：轻量预约演变为完整日程系统。
   - 缓解：本期只支持冲突场景下的稍后提醒或到空闲再尝试。

5. **恢复失败残留风险**：会议失败后 Agent 保持错误占用或任务无法恢复。
   - 缓解：恢复状态持久化，失败可见并提供人工补救路径。

## 进入 Checklist 条件

- 产品层面无阻塞问题。
- 技术层面风险可通过状态机、原子占用、幂等恢复和可见降级处理。
- 可进入 Phase 5 checklist 确认阶段。
