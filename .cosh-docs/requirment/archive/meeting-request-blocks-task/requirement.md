# Meeting Request Blocks Task

## 背景

项目执行中，部分任务在执行过程中会发现需求歧义、方案不一致或跨 Agent 无法单独决策的问题。此时 AI 会发起会议申请，希望通过 AI 会议达成统一后再继续任务。

当前系统中，AI 会议申请展示在任务详情和会议申请队列中，但不会阻塞项目任务执行。用户确认产品语义后，会议申请不再是普通协作提醒，而是任务继续执行的前置条件。

## 目标用户

- 使用 Project Execution 自动执行项目任务的用户。
- 在任务执行中依赖多个 AI Agent 协作决策的用户。
- 需要判断任务为何暂停、如何恢复、是否继续推进的项目负责人。

## 产品目标

- 当任务发起 AI 会议申请时，阻止任务在未达成一致前继续执行。
- 让任务在等待会议期间保持 In Progress 语义，并明确展示“等待会议结论”。
- 会议结论明确“已达成一致，可以继续执行”后，任务才自动恢复执行。
- 会议无共识时任务进入 Blocked。
- 会议申请被拒绝或超时未确认时，任务继续等待用户决策，而不是自动继续。
- 一个任务同一时间只允许一个阻塞型会议申请。

## 已澄清结论

- 凡是任务需要开会，说明存在歧义或不一致；会议申请一定是为了解决某个不一致问题。
- 会议申请应阻塞当前任务。
- 阻塞期间任务保持或回到 In Progress 产品语义。
- 会议结束后的恢复规则由会议结果决定：
  - 明确达成一致并允许继续：自动恢复执行。
  - 没有达成一致：进入 Blocked。
  - 需要人工判断：等待用户处理。
- 等待期间用户可以接管：取消等待并继续执行、标记阻塞、或重新申请会议。
- 会议申请被拒绝时，任务详情应显示“会议申请被拒绝，等待用户决定继续/重新申请/标记阻塞”。
- 同一个任务同一时间只允许一个阻塞型会议申请。
- 成功标准优先是防止 AI 在未达成一致时继续推进关键任务。

## 范围

本期必须覆盖：

- 任务会议申请创建后，项目任务进入等待会议结论状态。
- Project Execution 调度不再继续推进该任务，也不拉取后续 backlog 任务越过它。
- 会议申请确认、拒绝、会议完成、会议无共识等结果能反映回任务状态。
- 用户在任务详情中能看懂等待原因，并能做必要接管操作。
- 会议申请状态、任务执行状态、项目 workflow 状态保持一致。
- 中英文文案都可用。
- 会议申请队列需要适合大量申请的浏览方式：
  - 第一排序维度是处理状态，未处理申请优先于已处理申请；第二排序维度是更新时间/创建时间。
  - 申请列表只展示摘要信息，详细配置、上下文和处理动作通过 VO 风格弹窗展示。
  - 不同会议申请状态需要有不同颜色标识，便于快速区分待确认、已确认和已拒绝。
  - 侧边栏的待处理会议申请数量需要用同一行右侧小气泡展示，减少行距并提升辨识度。

## 非目标

- 不重新设计整个会议系统。
- 不改变普通非项目会议的行为。
- 不要求所有会议都能自动理解自由文本结论；只要求项目阻塞型会议有可判定的产品结果。
- 不新增复杂多会议编排；同一任务同一时间只允许一个阻塞型会议申请。

## 关键约束

- 不能让任务因为会议申请被拒绝或超时而自动继续。
- 不能让已处于等待会议结论的任务被项目连续执行流绕过。
- 不能让 AI 代替用户做“无共识时继续执行”的决策。
- 必须保留任务历史和会议申请记录，便于追踪为什么暂停或恢复。
- 会议申请列表 UI 不能因为详情内容过长而撑高队列；处理详情应在弹窗内完成。
