# Meeting Request Blocks Task Checklist

确认状态：已确认

## 验收与测试清单

### CHK-001 会议申请创建会阻塞当前任务

- 关联需求点：会议申请是任务继续执行的前置条件。
- 验证方法：在一个正在 Project Execution 执行的任务上创建 AI 会议申请。
- 预期结果：任务进入等待会议结论状态，仍保留 In Progress 产品语义；任务详情显示等待会议结论。

### CHK-002 Pipeline 不继续推进等待会议的任务

- 关联需求点：未达成一致前不能继续执行任务。
- 验证方法：让任务进入等待会议结论状态后观察 Project Execution workflow。
- 预期结果：当前任务不继续执行、不进入 review/done；项目不会越过该任务拉取后续 backlog 任务。

### CHK-003 同一任务只能存在一个未解决阻塞型会议申请

- 关联需求点：一个任务同一时间只允许一个阻塞型会议申请。
- 验证方法：对同一任务连续创建两次会议申请。
- 预期结果：第二次创建被拒绝或返回现有未解决申请，不产生多个竞争解锁任务的申请。

### CHK-004 会议确认后任务仍等待会议结果

- 关联需求点：会议申请确认不等于达成一致。
- 验证方法：确认会议申请并创建会议，但不结束会议。
- 预期结果：任务继续显示等待会议结论，不自动恢复执行。

### CHK-005 会议明确达成一致后任务恢复执行

- 关联需求点：会议产出“已达成一致，可以继续执行”后恢复。
- 验证方法：完成关联会议，并设置结构化结果为可继续。
- 预期结果：任务解除会议等待状态，恢复到可执行流程，并记录恢复原因和会议引用。

### CHK-006 会议无共识后任务进入 Blocked

- 关联需求点：没有达成一致时不得继续推进。
- 验证方法：完成关联会议，并设置结构化结果为无共识。
- 预期结果：任务进入 Blocked，阻塞原因包含会议无共识信息，workflow 停止推进。

### CHK-007 会议需要人工判断时任务等待用户处理

- 关联需求点：会议结果无法自动判断时用户接管。
- 验证方法：完成关联会议，并设置结构化结果为需要用户决定。
- 预期结果：任务保持等待用户处理状态，UI 提供继续执行、标记阻塞、重新申请会议入口。

### CHK-008 会议申请被拒绝不会自动继续

- 关联需求点：拒绝会议不代表歧义解决。
- 验证方法：创建会议申请后拒绝，并填写拒绝原因。
- 预期结果：任务显示“会议申请被拒绝，等待用户决定继续/重新申请/标记阻塞”，不自动恢复执行。

### CHK-009 会议确认或准备超时不会自动继续

- 关联需求点：超时不代表达成一致。
- 验证方法：构造 pending 或 preparing 超时的会议申请/会议。
- 预期结果：任务保持等待用户处理，不自动执行；UI 显示超时或需处理提示。

### CHK-010 用户可以取消等待并继续执行

- 关联需求点：等待期间用户可以接管。
- 验证方法：在等待会议结论状态点击继续执行。
- 预期结果：任务恢复执行流程，并在任务历史中记录人工 override 和操作者。

### CHK-011 用户可以将等待会议任务标记为 Blocked

- 关联需求点：用户可将无法解决的问题明确阻塞。
- 验证方法：在等待会议结论状态选择标记阻塞并填写原因。
- 预期结果：任务进入 Blocked，原因保存并展示；workflow 不继续推进。

### CHK-012 用户可以重新申请会议

- 关联需求点：被拒绝或无效会议后可重新处理。
- 验证方法：在等待用户处理状态重新申请会议。
- 预期结果：旧未解决会议阻塞被替换或关闭，新申请成为当前阻塞来源。

### CHK-013 任务状态和项目 workflow 状态一致

- 关联需求点：任务详情、看板和 workflow 状态不能互相矛盾。
- 验证方法：分别检查任务详情、任务卡片、项目执行状态条和 API 返回。
- 预期结果：均显示等待会议结论或对应终态，无一个位置显示仍在正常执行。

### CHK-014 中英文文案完整

- 关联需求点：UI 需要支持中文/英文。
- 验证方法：切换中文和英文界面，查看任务状态、会议申请区、用户操作按钮、toast/弹窗。
- 预期结果：不出现裸 key，不混用错误语言。

### CHK-015 普通非项目会议不受影响

- 关联需求点：不重新设计整个会议系统。
- 验证方法：创建普通会议或非项目任务会议。
- 预期结果：普通会议仍按原流程执行，不产生项目任务阻塞状态。

### CHK-016 非会议阻塞流程不回归

- 关联需求点：已有 Project Execution 状态机不能被破坏。
- 验证方法：执行任务通过 review、用户验收、拒绝返工、标记 blocked 等既有流程。
- 预期结果：既有流程保持可用，状态转移符合之前验收结果。

### CHK-017 真实数据端到端验证

- 关联需求点：必须能在真实项目中验收。
- 验证方法：创建或选择真实项目任务，触发会议申请、确认会议、分别验证可继续/无共识/拒绝路径。
- 预期结果：数据文件、API、UI 与任务状态一致，刷新页面后状态仍正确。

### CHK-018 可观测性与历史记录

- 关联需求点：用户需要知道为什么暂停或恢复。
- 验证方法：检查任务 comments/history/activity 或 task file。
- 预期结果：记录会议申请 ID、会议 ID、状态变化、用户 override、阻塞原因或恢复原因。

### CHK-019 会议申请列表按处理状态优先排序

- 关联需求点：会议申请队列需要先突出未处理申请，再按时间排序。
- 验证方法：构造 pending、confirmed、rejected 多条会议申请，且已处理申请时间晚于 pending 申请。
- 预期结果：pending 申请始终排在 confirmed/rejected 之前；同一状态组内按更新时间或创建时间倒序。

### CHK-020 会议申请详情通过 VO 风格弹窗展示

- 关联需求点：申请列表只展示摘要，详细信息和处理动作通过弹窗展示。
- 验证方法：打开 AI 会议申请队列，点击申请卡片或“查看详情/处理”按钮。
- 预期结果：列表卡片不再铺开完整配置；弹窗展示目标、期望产物、原因、上下文候选、参与人、主持人、确认/拒绝操作，并支持 Esc/遮罩关闭。

### CHK-021 会议申请状态颜色一致

- 关联需求点：不同会议状态需要用颜色标识，提高辨识度。
- 验证方法：分别查看 pending、confirmed、rejected 申请在会议队列和项目任务详情中的展示。
- 预期结果：待确认使用黄/橙色，已确认使用绿色，已拒绝使用红色；badge 与卡片边框/背景风格一致。

### CHK-022 侧边栏待处理申请数量使用同一行气泡

- 关联需求点：减少行距并提升待处理数量辨识度。
- 验证方法：存在待确认会议申请时查看左侧会议区域提示。
- 预期结果：提示文案和数量在同一行；数量位于最右侧并显示为小气泡，不再单独占第二行。
