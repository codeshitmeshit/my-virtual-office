# VO Usage Ledger 需求

## 背景

Virtual Office 现有 `API Usage` 面板主要读取 OpenClaw/provider 凭据对应的账号额度或 quota 状态。它能帮助判断账号是否可用、额度窗口是否接近限制，但并不能准确表达 VO 自己在聊天、项目执行、会议、定时任务等流程中实际消耗了多少 token。

当前用户容易把“账号额度剩余”误解为“VO 实际用量统计”，导致对统计准确性产生困惑。因此需要新增一套 VO 内部用量统计，保留旧面板但重新明确语义。

## 目标用户

主要用户是 VO 管理者。

他们关心：

- 今天 VO 总共用了多少 token。
- 哪些 Agent 用量最高。
- 哪些模型用量最高。
- 当前统计覆盖度是否可信，尤其是有多少 run 没有返回 usage。

## 产品目标

第一版目标是建立可信、清晰的 VO 内部用量观察入口。

它需要回答：

- VO 今天实际记录到了多少 token 消耗。
- 各 Agent 分别消耗多少。
- 各模型分别消耗多少。
- 最近哪些 run 有消耗记录，哪些 run 没有返回 usage。
- 统计覆盖度如何，避免用户误以为总量是完整精确账单。

## 范围

第一版包含：

- 保留旧 `API Usage` 能力，但产品表达改为 `Account Limits` / 账号额度。
- 新增 `VO Usage` 视图，与 `Account Limits` 放在同一区域，通过 tab 区分。
- 新增 VO 内部用量账本，记录 provider 已经返回的 usage/tokenUsage。
- 原始记录粒度为每次 Agent run 一条记录。
- 支持按 Agent 聚合。
- 支持按模型聚合。
- 支持展示今日总量、run 数、未统计 run 数和缺失率。
- 支持展示最近 run 列表。
- 对未返回 usage 的 run 不计入 token 总量，但需要在统计覆盖度或最近记录中体现。

## 非目标

第一版不包含：

- 不做额外模型调用询问 Agent “用了多少 token”。
- 不用估算值冒充真实 token。
- 不做成本金额估算。
- 不做完整项目/任务/会议成本归因。
- 不引入新的数据库作为前置要求。
- 不删除旧 `API Usage` 功能。
- 不做复杂图表、导出 CSV 或告警规则。

## 已澄清结论

- 旧面板应改名或表达为 `Account Limits`，含义是外部账号额度状态。
- 新面板命名为 `VO Usage`，含义是 VO 内部实际记录到的用量账本。
- UI 入口采用同一区域两个 tab：`Account Limits` 和 `VO Usage`。
- 第一版必须展示模型维度，即同时包含 `byAgent` 和 `byModel`。
- 无 usage 的 run 不计入 token 总量，但要显示未统计 run 数或缺失率。
- 成功标准优先是可信和不混淆，而不是覆盖所有维度或计算成本。

## 关键约束

- 用量记录必须来自 provider 已经返回的机器字段，例如 `usage` 或 `tokenUsage`。
- 对 usage 缺失的 run，第一版应标记缺失，不应估算。
- 原始账本应保留 run 粒度，以便后续扩展按项目、任务、会议、来源等维度聚合。
- 存储方式应贴近现有 VO 本地 JSON/JSONL 风格，避免引入过重依赖。
- 查询和 UI 展示不能高频全量扫描大量历史文件。

## 成功标准

- 管理者能清楚区分账号额度状态和 VO 实际用量。
- 管理者能看到今日 VO usage 总量、按 Agent 排名、按模型排名。
- 管理者能看到统计覆盖度，例如未返回 usage 的 run 数或缺失率。
- 旧 `Account Limits` 功能仍可用。
- 新统计不会因为缺失 usage 而给出误导性的完整总量。
