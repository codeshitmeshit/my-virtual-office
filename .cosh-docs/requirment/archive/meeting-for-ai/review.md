# Meeting for AI 需求与方案评审

## 评审结论

**结论：通过，可进入测试 checklist 确认阶段。**

产品目标、会议流程、控制权、并发边界和会后产物已经明确。现有代码具备可复用基础，但当前会议实现只是状态与记录层，需要新增独立的会议编排能力。

## 分期评审

需求拆分为七个可独立验收的 Phase：

| Phase | 交付重点 | 主要风险 | 进入条件 |
| --- | --- | --- | --- |
| Phase 1 | 领域模型、状态机、持久化、兼容 | 状态设计不足导致后续返工 | 无 |
| Phase 2 | 用户发起、顺序发言、有限讨论、结构化结果 | Provider 返回不一致、讨论重复 | Phase 1 |
| Phase 3 | 用户干预、暂停继续、争议裁决、主持接管 | 并发事件造成状态错乱 | Phase 2 |
| Phase 4 | AI 申请、用户确认、上下文快照 | 未授权上下文泄漏、会议滥用 | Phase 2 |
| Phase 5 | 忙碌冲突、任务暂停恢复、会议并发 | Provider 暂停语义不一致 | Phase 1-3 |
| Phase 6 | 行动项确认与项目任务闭环 | 重复创建、错误负责人 | Phase 2，正式验收依赖 Phase 5 |
| Phase 7 | 重启恢复、幂等、日志、安全、回归 | 异常组合多、测试成本高 | Phase 1-6 |

### Phase 1 评审重点

- 状态机和事件序号必须先稳定，不能直接以聊天消息作为会议状态。
- 持久化数据应支持原子更新或等价保护。
- 旧会议只能作为兼容记录，不应伪造执行阶段。

### Phase 2 评审重点

- MVP 只接受空闲 Agent，主动缩小首期任务暂停风险。
- 会议推进必须由服务端编排，浏览器关闭后仍能继续。
- 每次 Agent 调用必须绑定会议 ID、阶段、轮次和发言序号。

### Phase 3 评审重点

- 用户干预需要进入同一事件队列，不能直接修改内存变量。
- 暂停操作必须阻止创建新的 Agent 调用。
- 主持人更换后不能重放已完成发言。

### Phase 4 评审重点

- 申请和正式会议必须是不同状态。
- 上下文候选与最终快照必须分离。
- 用户删除的候选资料不得出现在调用日志、发言或总结中。

### Phase 5 评审重点

- 会议占用校验必须是原子的。
- 任务暂停和恢复需要 provider 能力矩阵。
- 逻辑暂停不能被描述为真实进程暂停。
- 恢复必须具备幂等标识和人工补救状态。

### Phase 6 评审重点

- 行动项草稿与正式项目任务必须是两个对象。
- 用户确认请求需要幂等键。
- 项目任务保留来源会议和原始行动项，便于追踪。

### Phase 7 评审重点

- 重启恢复测试必须覆盖每个关键阶段。
- 原始发言、滚动摘要和结构化结果需要可交叉核对。
- 日志不得泄露未确认上下文或 provider 凭据。

## Phase Gate

每个 Phase 进入下一阶段前必须满足：

1. 本 Phase 对应 checklist 全部通过。
2. 新增状态和接口具备回归测试。
3. 不破坏旧会议展示、普通聊天和项目工作流。
4. 失败场景具有可理解状态，不留下无法清理的 Agent 占用。
5. 文档记录本 Phase 的已知限制，不能把后续 Phase 能力描述为已实现。

## 当前能力评估

### 已有能力

- `POST /api/meetings/create` 可创建会议记录。
- `GET /api/meetings/active` 和历史接口可查询会议。
- `POST /api/meetings/end` 可保存摘要、决议、行动项和 Agent 回应。
- 前端可以展示会议、参与人、历史和结束表单。
- `processMeetings()` 可驱动 Agent 进入会议状态并移动到会议区域。
- 跨平台 Agent 通信已经支持稳定 `conversationId`。
- 项目系统已经支持创建待办任务和分配 Agent。
- Codex、OpenClaw、Hermes 已有各自的调用或通信适配层。

### 缺失能力

- 没有会议申请和用户确认状态。
- 没有主持人、阶段、轮次、发言顺序和结束条件的执行逻辑。
- 创建会议不会通知或调用参会 Agent。
- 没有共享会议上下文和跨 Agent 发言转发。
- 没有用户实时干预会议的统一入口。
- 没有参会 Agent 忙碌冲突与会议占用锁。
- 没有暂停、恢复原任务的可靠机制。
- 没有超时、离线、失败和服务重启恢复。
- 会后行动项未与待确认项目任务形成闭环。

## 推荐方案边界

### 1. 会议领域对象

现有会议对象应升级为持久化会议实体，至少包含：

- 基础信息：ID、主题、目的、类型、发起者、主持人。
- 生命周期：申请中、待确认、准备中、进行中、暂停、等待裁决、结束、取消、失败。
- 流程信息：当前阶段、当前轮次、最大轮次、当前发言者、发言队列。
- 参与者状态：已邀请、等待、参会、忙碌、离线、失败、退出。
- 上下文：候选上下文、用户确认后的上下文快照。
- 讨论记录：结构化发言事件和用户干预事件。
- 结果：摘要、决议、争议、行动项和贡献。
- 恢复信息：参会前状态、原任务和恢复结果。

### 2. 会议编排器

新增独立会议编排器负责：

1. 校验申请和参会冲突。
2. 锁定参会 AI 的会议占用。
3. 保存并暂停原任务状态。
4. 依次执行首轮陈述。
5. 根据主持策略执行有限轮次讨论。
6. 接收用户干预并调整后续议程。
7. 判断完成、等待裁决、超时或失败。
8. 生成结构化结果。
9. 恢复 Agent 原任务并释放占用。

编排器应以持久状态推进，不能只依赖单个 HTTP 请求或内存线程。

### 3. Agent 调用适配

- 为不同 provider 复用现有调用能力。
- 每个会议使用稳定的会议会话标识。
- 每次发言输入应包含已确认上下文、当前议题、相关发言摘要、主持指令和剩余轮次。
- Agent 返回内容需要标准化为观点、依据、异议、建议和是否需要追问。
- 单个 Agent 失败不能直接破坏整场会议，主持人应记录失败并按规则继续或暂停。

### 4. 用户控制

用户操作必须作为会议事件持久化，包括：

- 确认或拒绝 AI 会议申请。
- 编辑会议配置。
- 发言、追问和点名。
- 调整议题或补充上下文。
- 暂停、继续、取消和结束。
- 对争议做出裁决。
- 确认或拒绝会后任务。

### 5. Agent 占用和任务恢复

- 同一个 Agent 必须具有排他的会议占用标识。
- 强制加入仍需记录原任务状态和中断原因。
- 恢复操作必须幂等，避免任务被重复启动。
- 若原 provider 不支持可靠暂停，应将状态标记为“逻辑暂停”，阻止系统继续派发新工作，并明确展示限制。

### 6. 行动项转任务

- 会后行动项先保存为待确认草稿。
- 用户确认后才调用项目任务创建能力。
- 每个任务必须保留来源会议 ID、原始行动项、建议负责人和确认人。
- 重复确认不得重复创建任务。

## 状态流建议

```text
draft/requested
  -> pending_confirmation
  -> preparing
  -> active_opening
  -> active_discussion
  -> awaiting_user_decision
  -> summarizing
  -> completed
```

异常分支：

```text
preparing/active -> paused -> active
preparing/active -> cancelled
preparing/active -> failed
```

## 兼容性与迁移

- 现有 `_meetings` 和 `_meetingHistory` 数据需要兼容读取。
- 旧会议可按“仅记录型会议”展示，不要求补齐执行事件。
- 现有 API 可以保留，但创建接口应区分“记录会议”和“执行会议”，或统一升级并提供兼容默认值。
- 当前办公室会议动画继续由会议参与状态驱动。

## 权限与安全

- AI 发起会议必须经过用户确认。
- 用户是最高权限控制者。
- 会议上下文必须限制在用户确认的资料范围内。
- 发言记录和会后结果不得包含未授权工作区或其他会话内容。
- 强制打断 Agent、取消会议和创建任务必须留审计记录。

## 异常处理

- Agent 忙碌：进入冲突处理，不自动打断。
- Agent 离线：允许更换、移除、等待或取消。
- Agent 超时：有限重试后记录缺席，主持人决定继续或暂停。
- 主持 AI 失败：允许用户接管或更换主持人。
- 服务重启：从持久化阶段和事件序号恢复，不重复已完成发言。
- 用户长时间未裁决：保持等待状态，不自动选择方案。
- 任务恢复失败：会议仍可结束，但必须显示恢复失败和人工处理入口。

## 性能与并发

- 多场会议可以并行。
- 单 Agent 会议占用必须使用原子校验。
- 每场会议的发言推进串行化，避免同一轮重复发言。
- 讨论记录应支持增量读取，避免前端反复加载全部历史。

## 可观测性

应记录：

- 会议状态变化。
- 每轮开始和结束。
- Agent 调用耗时、超时和错误。
- 用户干预事件。
- 占用锁和任务暂停、恢复结果。
- 结果生成和任务确认结果。

## 风险

1. **伪讨论风险**：多个 AI 可能重复相似观点。
   - 通过独立首轮、主持人摘要和针对性追问降低重复。
2. **上下文膨胀**：完整发言历史可能快速变长。
   - 使用结构化事件和滚动摘要，但保留原始发言供审计。
3. **任务暂停能力不一致**：不同 provider 的中断语义不同。
   - 产品层统一展示，适配层明确实际暂停或逻辑暂停。
4. **自动总结失真**：主持 AI 可能遗漏少数意见。
   - 结果必须保留争议和每位 AI 的贡献，用户可编辑后确认。
5. **会议滥用**：AI 可能频繁申请会议。
   - 所有 AI 发起均需用户确认，并展示申请理由与预期产物。

## 非阻塞技术决策

以下细节可在开发任务拆解阶段按现有代码模式确定，不阻塞 checklist：

- 会议持久化文件拆分方式。
- 前端采用轮询、现有 WebSocket 或增量事件接口。
- 主持提示词和结构化返回格式的具体字段。
- 各 provider 的暂停能力降级策略。

## 评审结论

该需求不应在现有 `_handle_meeting_create()` 上直接堆叠同步调用。应保持现有展示兼容，同时引入持久化会议状态机和可恢复编排器。当前无阻塞性产品或技术问题，可以进入验收 checklist 确认。
