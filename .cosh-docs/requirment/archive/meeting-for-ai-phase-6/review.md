# Meeting for AI Phase 6 方案评审

## 评审结论

**结论：通过，可进入测试 checklist 确认阶段。**

Phase 6 的产品边界已经明确：它不是自动执行会议结论，也不是重做项目管理系统；它是在会议结果和项目任务之间建立一个用户可确认、可编辑、可追踪、幂等的转换层。会议绑定项目是本 Phase 的必要前置能力，因为正式任务必须有项目归属。

## 产品评审

### 已清晰的产品假设

- 用户拥有最终确认权。行动项不能自动变成正式任务。
- 会议可以绑定项目，但不是所有会议都必须绑定项目。
- 已绑定项目的会议默认把行动项草稿归到该项目。
- 未绑定项目的会议在确认任务前必须选择目标项目、新建项目，或只保存为会议行动项。
- 新建项目和创建正式任务都需要用户显式确认。
- 行动项草稿与正式任务是两个不同对象。

### 仍需在实现中保持的产品边界

- 不要把行动项草稿显示成“已创建任务”。
- 不要让 AI 主持人或 advisory 建议直接创建项目任务。
- 不要在没有目标项目时静默创建任务。
- 不要把拒绝行动项等同于删除会议结果；拒绝记录仍需保留。
- 不要把本 Phase 扩展成完整日历、排期或项目管理重构。

## 技术评审

### 数据模型

- 会议实体需要支持可选 `projectId`。
- 会议结果需要包含稳定的 action item ID。
- 行动项草稿需要独立持久化，至少保存来源会议、目标项目、建议负责人、编辑后内容、状态和审计字段。
- 正式项目任务需要保存 `sourceMeetingId`、`sourceActionItemId` 和原始行动项快照，避免后续会议结果变更破坏追溯。

### 状态流

- 推荐状态流：`draft` -> `confirmed` / `rejected` / `kept_as_meeting_item`。
- 确认前允许编辑；确认后应禁止再次创建任务，只允许查看关联任务。
- 项目绑定变更应只影响未确认草稿；已确认任务不应被静默迁移。
- 未绑定项目的确认请求必须被服务端拒绝，除非请求明确选择新建项目或仅保存为会议行动项。

### 幂等与一致性

- 任务确认接口必须支持幂等键，或使用 `(meetingId, actionItemId)` 唯一约束实现等价保护。
- 重复确认同一行动项应返回同一个已创建任务，而不是创建新任务。
- 前端重复点击、网络重试、页面刷新后的状态恢复都要覆盖。

### 权限与审计

- 项目绑定、任务确认、任务拒绝和新建项目都属于需要留痕的用户操作。
- 审计记录应包含操作者、时间、动作、变更前后状态、会议 ID、行动项 ID 和任务 ID。
- 如果项目权限模型存在限制，确认任务时必须校验用户是否能向目标项目创建任务。

### UI 与集成

- 会议创建、AI 申请确认和会议结果页都需要体现项目绑定。
- 会议结果页需要支持行动项列表、编辑、确认、拒绝、选择项目、创建后跳转。
- 项目任务详情需要展示来源会议入口。
- 普通项目任务创建入口不能被 Phase 6 改动破坏。

## 主要风险

1. **自动执行风险**：会议行动项被自动创建为正式任务。
   - 缓解：草稿与正式任务分离，确认前项目中不存在正式任务。

2. **归属不明风险**：未绑定项目的行动项被创建到错误项目或默认项目。
   - 缓解：服务端强制要求目标项目、新建项目确认或仅保存为会议行动项。

3. **重复创建风险**：用户重复点击确认或请求重放产生多个任务。
   - 缓解：确认接口使用幂等键或唯一约束，重复确认返回同一任务。

4. **追溯丢失风险**：项目任务无法追溯来源会议和原始行动项。
   - 缓解：正式任务保存来源会议、来源行动项和原始行动项快照。

5. **范围膨胀风险**：项目绑定演变为完整项目管理重构。
   - 缓解：本期只做会议项目绑定、行动项草稿、确认建任务和双向链接。

## 进入 Checklist 条件

- 产品层面无阻塞问题。
- 技术层面风险可通过数据模型分离、服务端校验、幂等确认、审计记录和 UI 状态约束处理。
- 可进入 Phase 6 checklist 确认阶段。
