# Preparing Meeting Timeout Release

## 背景

当前会议中心支持可执行会议。用户发起会议后，会议会进入 `preparing` 准备中状态，截图中的会议卡片显示为“可执行会议 · 准备中”。现有代码中，准备态会议创建时会为参会 Agent 建立占用；如果用户长时间没有点击开始、取消或处理后续动作，准备态会议可能持续占用 Agent，影响后续会议和任务调度。

用户希望：准备中的会议如果 5 分钟没有开始，系统支持超时释放；释放时间可在设置中配置，单位为秒。

## 目标用户

- 普通使用者：发起 AI 会议后忘记启动或离开页面，希望系统自动清理无效准备态。
- 会议主持人：需要可配置的准备态保留时长，避免误释放仍在准备中的会议。
- 系统维护者：需要会议占用状态不会因遗留准备态而长期阻塞。

## 目标

1. 为可执行会议的 `preparing` 状态增加自动超时释放机制。
2. 默认释放时间为 300 秒，匹配用户提出的 5 分钟期望。
3. 在设置中提供秒级配置项，允许用户调整准备态超时时间。
4. 超时释放必须释放参会 Agent 的会议占用，避免后续会议被误判冲突。
5. 超时释放要有可追踪的状态、事件或历史信息，用户能理解会议为什么不再处于准备中。

## 范围

- 后端会议状态：识别超时的 `preparing` 可执行会议，并转入终态或释放态。
- 占用释放：清理 `store["occupancy"]` 中对应参会 Agent 的占用。
- 设置项：在现有设置界面增加“准备中会议超时释放（秒）”配置。
- 配置持久化：把设置保存到现有 `vo-config.json` 或现有 display/preferences 配置结构中，并由后端读取。
- 前端展示：准备态会议显示剩余时间或超时释放后的状态说明。
- 测试：覆盖默认 300 秒、用户配置、超时释放、非准备态不释放和回归路径。

## 非目标

- 不改变会议正常启动后的讨论、暂停、等待用户决策、总结、完成等流程。
- 不引入复杂调度系统；可以在会议投影、reconcile、run/start 等入口进行惰性清理。
- 不自动启动会议。
- 不自动处理冲突态会议；冲突态有独立的冲突处理语义。
- 不删除会议记录；释放后仍应能在历史或详情中看到原因。

## 关键约束

- 单位必须是秒。
- 默认值应为 300 秒。
- 需要兼容既有会议数据，旧会议缺少新字段时不能报错。
- 释放动作必须幂等，重复扫描不能重复追加错误事件或导致占用异常。
- `preparing` 会议如果已经被用户启动、取消、完成或进入其他阶段，不应被准备态超时逻辑处理。
- 设置值需要做边界校验，避免负数、非数字或过小值造成立即误释放。

## 当前已知代码结论

- 可执行会议阶段定义在 `app/server.py`，包含 `draft`、`conflict`、`preparing`、`active_opening`、`active_discussion`、`paused`、`awaiting_user_decision`、`summarizing`、`completed`、`cancelled`、`failed`。
- `_handle_executable_meeting_create` 在没有冲突时创建 `preparing` 会议，并把参会者写入 `store["occupancy"]`。
- `_handle_executable_meeting_transition` 在终态时释放占用。
- `_meeting_active_projection` 返回活动会议列表，适合作为惰性清理入口之一。
- `_handle_executable_meeting_reconcile` 已经存在会议占用重建入口，适合作为显式修复入口。
- 前端设置页已有秒级配置项示例：内部状态气泡自动收起。
- 前端会议详情在 `app/game.js` 中对 `preparing` 状态展示“Start meeting”按钮。
