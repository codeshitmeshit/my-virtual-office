# 项目定时任务 Phase 4：执行历史与控制面板异常提示

## 背景

项目定时任务 Phase 1 已完成 Cron 项目绑定基础能力，Phase 2+3 已完成项目详情页配置、全局定时任务总览和项目执行派发。当前系统已经可以创建、编辑、立即运行、暂停/恢复和派发项目定时任务。

Phase 4 需要解决的是执行后的可见性问题：用户需要知道项目定时任务是否运行、为什么跳过、哪里失败，以及哪些异常需要处理。正常运行不应打扰控制面板，但失败和需要人工介入必须可见。

## 目标用户

- 项目负责人或项目创建者：需要在项目详情页追踪项目定时任务历史。
- 系统操作者：需要在控制面板看到需要处理的项目定时任务异常。
- 开发和排障人员：需要通过诊断日志定位项目定时任务的调度决策和失败原因。

## 目标

- 在项目详情页展示项目定时任务执行历史。
- 保存每次调度决策的核心上下文，包括时间、cron、项目、目标、状态和原因。
- 将正常执行、正常跳过和暂停记录在项目历史中，但不制造控制面板噪音。
- 只把失败和需要人工介入的状态提示到控制面板。
- 控制历史记录规模，保证长期项目运行后页面仍然可读、可用。
- 补充服务端诊断信息，便于排查调度问题。

## 范围

### 本子需求包含

- 设计并持久化项目定时执行历史记录。
- 在项目详情页展示执行历史列表。
- 历史记录覆盖 started、skipped、paused、failed、completed、intervention_required 等状态。
- skipped 记录需要展示用户可理解的原因，例如 workflow 已运行、项目暂停、任务缺失、重复触发未开启。
- 控制面板只提示 failed 和 intervention_required。
- 控制面板异常提示需要能定位到项目和定时任务。
- 历史记录需要有数量上限或截断策略，避免长期增长拖慢项目页。
- 服务端日志或诊断输出需要包含 projectId、cronId、targetType、taskId、decision、reason、error 和 timestamp。
- 补充自动化测试和 8090 live 验收记录。

### 本子需求不包含

- 不实现自动重试策略。
- 不实现复杂通知偏好设置。
- 不实现外部消息通知。
- 不改造通用控制面板架构为告警平台。
- 不改变 Phase 2+3 已完成的创建、编辑、派发和重复触发语义。
- 不处理 Phase 5 的长期项目推荐和完整本地化收尾。

## 关键产品规则

- 正常 started/completed 只进入项目历史，不进入控制面板提示。
- 正常 skipped 只进入项目历史，不进入控制面板提示，例如 project_active、paused、repeat disabled。
- failed 必须进入项目历史，并在控制面板出现异常提示。
- intervention_required 必须进入项目历史，并在控制面板出现人工介入提示。
- 控制面板提示应足够短，但必须提供项目名、定时任务名、目标和原因。
- 项目详情历史应足够完整，便于用户判断“这个定时任务到底发生了什么”。

## 依赖

- 依赖 Phase 1 的项目 cron 绑定存储和 CRUD API。
- 依赖 Phase 2+3 的项目详情页定时任务 UI、全局定时任务总览和派发逻辑。
- 依赖现有项目存储、项目活动记录、控制面板数据刷新机制。

## 成功标准

- 用户能在项目详情页看到最近的项目定时任务执行历史。
- 每条历史记录能说明什么时候、哪个定时任务、目标是什么、结果是什么、为什么。
- 失败和人工介入能出现在控制面板，并能定位回项目。
- 成功、正常跳过和暂停不污染控制面板。
- 长期运行后历史列表仍可读，存储不会无限增长。
- 自动化测试和真实 8090 验收覆盖主要状态。
