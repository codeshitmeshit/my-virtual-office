# Archive Room Requirement

## Background

Virtual Office already persists projects, tasks, meetings, cross-agent communications, Codex thread mappings, Hermes chat history, and workflow state under `VO_STATUS_DIR`. As project count, chat volume, AI participation, and generated artifacts grow, raw history alone is not enough for humans or newly joined AI agents to quickly understand a project's current state.

The requested product is an independent main-application module named "档案室" / "Archive Room". It should turn project activity, task results, key conversations, meetings, decisions, risks, and artifacts into a high-signal project archive that humans can browse and AI agents can load on demand.

## Product Goals

- Let humans quickly understand every project's current situation, risks, pending confirmations, and available artifacts.
- Let new AI agents join a project faster by reading a standard onboarding package instead of all raw chat history.
- Let active AI agents query concise, source-backed project context and load more detail only when needed.
- Preserve important project knowledge as durable assets rather than leaving it buried in chat, task logs, or meetings.
- Maintain accuracy through source links, freshness state, confidence levels, and human confirmation for long-lived rules.

## Primary Users

- Human project owner or operator using Virtual Office.
- New AI agents joining an existing project.
- Active execution AI agents working on project tasks.
- A global OpenClaw-based archive management AI responsible for maintaining project archives.

## Core Product Decisions

- Archive Room is a first-level navigation module in the main app, not only a project detail tab.
- The first screen is a project list / project overview view.
- The project list defaults to risk and pending-confirmation priority, with recent updates also considered.
- Archive Room stores high-signal project assets, not a full dump of every conversation.
- Project context is layered:
  - Project overview and current state.
  - Standard AI onboarding package.
  - Key decisions, risks, blockers, and long-lived rules.
  - Task artifacts and delivery records.
  - Source-backed timeline and raw-history references.
- The human UI and AI context API share the same underlying archive data, but present it differently:
  - Human UI is optimized for browsing, previewing, and confirming.
  - AI responses are structured context packages with conclusions, source references, and optional follow-up entries.
- The archive management AI is a single global OpenClaw agent.
- If the archive management AI does not exist, the system should automatically create it and show "已自动创建" in Archive Room.
- Users may pause and resume the archive management AI, but should not delete it from Archive Room in the first version.
- Archive management uses event-triggered updates plus scheduled inspection:
  - Run once after each VO startup.
  - Run once daily.
  - Run after important events such as task completion, meeting end, project state changes, user-marked important messages, AI stage summaries, blockers, important artifacts, and conflict alerts.

## Content Scope

### Included In Archive

- Project basic status:
  - Current stage.
  - Task counts.
  - Completion rate.
  - Risk count.
  - Pending confirmation count.
  - Active AI agents.
  - Last updated time.
- Project knowledge:
  - Goals.
  - Current state.
  - Key rules.
  - Key decisions.
  - Risks and blockers.
  - Important context.
  - Outdated context marked as outdated rather than deleted.
- Task artifacts:
  - Task completion reports.
  - Implementation summaries.
  - Test results.
  - Delivery documents.
  - Associated files.
- Common media artifacts:
  - Documents readable in Archive Room.
  - Images previewable.
  - Videos playable.
  - Audio playable.
  - Other attachments downloadable or openable.
- Maintenance records:
  - Archive AI status.
  - Last整理 / inspection time.
  - Auto-created marker.
  - Pending confirmation queue.
  - Conflict and reminder history.

### Not First-Version Goals

- Replacing the project/task module.
- Replacing raw provider-native chat logs.
- Guaranteeing rich preview for every binary or compressed file type.
- Letting users delete the archive management AI from Archive Room.
- Making every chat message a first-class archive item.

## Trust And Governance

Archive entries use three confidence levels:

- Confirmed fact: human-confirmed or otherwise trusted long-lived project information.
- AI inference: generated by archive management AI from project activity and sources.
- Pending confirmation suggestion: visible to humans and AI, but AI must treat it as lower confidence.

Long-lived project rules require human confirmation, including:

- Architecture conventions.
- Business rules.
- Quality standards.
- Workflow/process rules.

Conflict handling:

- Confirmed facts cannot be automatically overwritten.
- AI inference and pending suggestions may be updated automatically when newer or stronger sources exist.
- Outdated information is kept but marked as outdated.
- Severe conflicts first notify the execution AI.
- If the execution AI continues violating confirmed content, escalate to the human project owner.

Reminder severity:

- Severe conflict: interrupt execution AI.
- Normal risk: show as a reminder.
- Low-value supplement: record silently without interrupting.

## Archive Management AI Behavior

- Global OpenClaw agent, default role: archive manager / 档案管理员.
- Automatically created when missing.
- Responsible for:
  - Maintaining project archive summaries.
  - Creating and refreshing onboarding packages.
  - Identifying important project events.
  - Recording task artifacts.
  - Marking stale context.
  - Detecting conflicts and reminding execution AI.
  - Preparing pending confirmation suggestions for humans.
- Default work rhythm:
  - Event-triggered整理 after important project events.
  - Startup inspection after VO starts.
  - Daily inspection.
- User controls:
  - View status.
  - View recent整理 records.
  - Pause.
  - Resume.

## Phase Plan

### Phase 1: Archive Data Foundation

Goal: establish durable archive records under the existing VO data model.

Scope:

- Define project archive files under `VO_STATUS_DIR`.
- Store project overview, confidence levels, source references, stale flags, pending confirmations, and artifact metadata.
- Derive initial project archive data from existing project/task records.
- Keep raw source references instead of duplicating all raw history.

Exit criteria:

- Every project can have a durable archive record.
- Archive data survives server restart.
- Archive records can distinguish confirmed fact, AI inference, and pending confirmation suggestion.

### Phase 2: Archive Room Main Navigation And Project Overview

Goal: make Archive Room visible as a first-level app module.

Scope:

- Add "档案室" to the main app navigation.
- Build the project overview list.
- Show project status metrics, risks, pending confirmations, active AI, last update, and artifact counts.
- Default sorting prioritizes risk / pending confirmation, then recent update.
- Show archive manager status, including auto-created, running, paused,整理中, missing, or error.

Exit criteria:

- Humans can open Archive Room from the main UI.
- Humans can understand project status across projects without opening each project.
- Archive manager status is visible.

### Phase 3: Project Archive Detail And Artifact Preview

Goal: let humans inspect one project's archive and artifacts.

Scope:

- Add project archive detail view.
- Show current state, key decisions, risks, blockers, long-lived rules, onboarding package, and timeline references.
- Show task artifacts and associated files.
- Support document reading, image preview, video playback, audio playback, and download/open behavior for other files.
- Show source links and confidence level per important entry.

Exit criteria:

- Humans can inspect a project's archive.
- Videos and audio artifacts can be played when file type/browser support allows.
- Artifact entries remain usable even when a file cannot be previewed.

### Phase 4: Archive Management AI Lifecycle

Goal: provide the global OpenClaw archive manager agent lifecycle.

Status: implemented and self-tested on 2026-06-20 in child requirement `archive-room-phase-4`; awaiting human acceptance.

Scope:

- Detect whether the global archive management AI exists.
- Automatically create it when missing.
- Surface "已自动创建" and current status in Archive Room.
- Allow pause and resume.
- Record recent maintenance activity.

Exit criteria:

- Missing archive manager is automatically created.
- Users can see and control its running/paused state.
- Archive Room still works in a degraded read-only way if creation fails.

### Phase 5: Event-Triggered And Scheduled整理

Goal: keep archives current without requiring manual maintenance.

Scope:

- Trigger整理 on task completion, meeting end, project state change, explicit important mark, AI stage summary, blocker, important artifact, and conflict reminder.
- Run inspection once after VO startup.
- Run inspection once daily.
- Detect important chat using both explicit marking and archive AI classification.
- Add整理 records and pending confirmation entries.

Exit criteria:

- Important project activity updates archive content.
- Daily/startup inspection fills gaps.
- Low-value activity does not flood the archive.

### Phase 6: AI Onboarding And Context Query

Goal: make archive data useful to AI agents.

Scope:

- Add a human-readable archive introduction for each project:
  - What this archive is for.
  - What information is currently inside it.
  - What will be added in the future.
  - What humans and AI agents can do with it.
- Show project basic information before mechanical summaries:
  - Project name, description, status, owner/participants when available, active AI, task progress, last update, maintenance state, artifact count, pending confirmation count, and major source types.
- Provide standard project onboarding package:
  - Project goal.
  - Current state.
  - Key rules.
  - Current task.
  - Key decisions.
  - Risks/blockers.
  - Related archive index.
- Provide project-characterized AI context injection:
  - Inject project-specific business background, confirmed rules, user preferences, decision style, important history, risks, and relevant artifacts into AI project/task context.
  - The injected context should make the same AI behave with awareness of the current project's characteristics.
  - This must not rewrite the AI's global identity, safety boundaries, or general tool rules.
- Provide context query behavior:
  - Return most relevant conclusions first.
  - Include source references.
  - Include optional entries the AI may load next.
- Support archive manager reminders to execution AI for conflicts, missing context, and stale decisions.

Exit criteria:

- Humans can open a project archive and immediately understand what the archive represents, what it contains, and how it can be used.
- New AI agents can start with a standard context package.
- AI agents working on different projects receive different project-characterized context, so their project/task behavior reflects the current project's goals, rules, preferences, risks, and artifact history.
- Execution AI can query Archive Room without reading all raw history.
- Severe conflicts can be surfaced AI-to-AI before escalating to humans.

### Phase 7: Confirmation Queue And Governance Polish

Goal: make accuracy governance practical for humans.

Scope:

- Add pending confirmation queue.
- Add a dedicated pending confirmation section in Archive Room project detail; it is not enough to show only the pending count on project cards.
- Let humans confirm or reject long-lived rules and high-impact suggestions.
- Preserve confirmation records.
- Show each pending item with proposed content, confidence, impact area, reason, source references, created time, and confirm/reject/defer actions.
- Show stale and conflicting entries clearly.
- Add filters and sort modes for risks, pending confirmations, recent updates, and active projects.

Exit criteria:

- Humans can resolve pending confirmations in Archive Room.
- Humans can open a project detail page, inspect all pending confirmation items, understand their evidence, and act on them without leaving Archive Room.
- Confirmed entries are protected from automatic overwrite.
- Project archive reliability is visible and manageable.

### Phase 8: Maintenance Frequency And Scheduling Controls

Goal: let users control how often Archive Room整理 runs for projects that need long-term maintenance.

Scope:

- Add per-project archive maintenance frequency configuration.
- Keep the existing long-term maintenance on/off setting, and add schedule options such as:
  - Event-triggered only.
  - Daily.
  - Weekly.
  - Custom interval, if implementation cost is reasonable.
- Show the next scheduled整理 time, last scheduled整理 time, and last event-triggered整理 time in Archive Room.
- Let users adjust the frequency from the Archive Room project detail UI.
- Persist the schedule under project archive maintenance state.
- Ensure paused archive manager or disabled project maintenance skips scheduled整理 with a clear maintenance record.
- Prevent duplicate scheduled整理 when VO restarts or multiple trigger paths fire close together.
- Optimize governance so the archive manager resolves most archive knowledge changes before involving humans:
  - Low-risk, source-backed整理 should be auto-confirmed by the archive manager.
  - Conflicts should first be classified by the archive manager as wording drift, stale old context, stronger new source, or true owner decision.
  - The archive manager may automatically mark old non-human-confirmed context as stale when newer stronger sources exist.
  - Human confirmation should be reserved for high-impact owner decisions, mutually conflicting high-trust sources, or changes that replace human-confirmed rules.
  - The pending-human queue should stay small, high-value, and actionable.

Exit criteria:

- Humans can see and change each long-term-maintained project's整理 frequency.
- Scheduled整理 follows the selected project frequency.
- Event-triggered整理 still works independently for important events.
- Disabled or paused maintenance does not run scheduled整理 and records the skip reason.
- Maintenance records clearly distinguish manual, event-triggered, startup, daily/weekly/custom scheduled整理.
- Archive manager handles ordinary archive governance automatically and only escalates truly ambiguous or high-impact decisions to humans.
- Pending-human items include the archive manager's decision summary, why automation was insufficient, and what human choice is needed.

## Success Criteria

- A new AI can quickly join a project by loading the standard onboarding package.
- A human can open Archive Room and understand project status, risk, pending confirmations, and major artifacts across projects.
- Important task artifacts and media are visible and usable from Archive Room.
- Archive manager operates as a visible global OpenClaw agent with automatic creation, status, pause, and resume.
- Archives update from important events and scheduled inspections without flooding users with low-value entries.
- Long-lived project rules and high-impact knowledge retain traceability and confirmation status.
