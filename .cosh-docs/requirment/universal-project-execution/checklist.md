# 通用项目执行 通用项目执行测试清单

确认状态：已确认

## 验收规则

- 通用项目执行 分为 执行底座 和 独立审查与最终验收，两阶段分别开发、测试和人工确认。
- 执行底座 通过后才能进入 独立审查与最终验收；独立审查与最终验收 通过后才代表 通用项目执行 整体达到“测试通过”。
- 执行底座 必过项：CHK-001 至 CHK-018、CHK-032 至 CHK-036，以及 CHK-038。
- 独立审查与最终验收 必过项：CHK-019 至 CHK-037，并回归 执行底座 的 CHK-005、CHK-008、CHK-010、CHK-015 至 CHK-018、CHK-032 至 CHK-036。
- 自动化测试应尽量使用隔离的临时状态目录、临时工作区和可控的 Provider 模拟实现。
- 真实 Provider 冒烟测试必须记录 Provider、Agent、工作区，以及是否使用演示或模拟模式。
- 两个阶段都必须经过浏览器人工验证，并分别记录用户确认。

## 执行底座：通用项目执行底座

执行底座 交付项目工作区绑定、角色配置、指定任务启动、三类 Provider 执行、证据收集、异常处理和恢复。执行成功后任务停在“执行完成”，不得调用旧版自审逻辑或进入 Done。

## 项目绑定与数据迁移

### CHK-001 旧项目数据迁移
- 验证方法：加载缺少 通用项目执行 字段的旧版 JSON 和 Markdown 项目，覆盖进行中任务及历史已完成任务。
- 预期结果：项目无数据丢失；缺少的工作区和默认角色字段获得兼容默认值；历史已完成任务不会被重新打开。
- 关联需求点：向后兼容升级；保护现有项目数据。

### CHK-002 绑定有效的 Git 工作区
- 验证方法：通过 API 和界面将项目绑定到可访问的 Git 仓库，然后重启服务并重新加载项目。
- 预期结果：规范化后的真实路径、`git` 类型、校验状态和项目默认角色均能正确持久化与恢复。
- 关联需求点：项目必须绑定本地目录或 Git 仓库；项目注册信息持久化。

### CHK-003 绑定有效的非 Git 目录
- 验证方法：绑定一个可访问但不是 Git 仓库的目录并执行校验。
- 预期结果：项目被识别为普通 `directory`；跳过 Git 专属检查，但不会被误判为无效绑定。
- 关联需求点：支持普通本地目录。

### CHK-004 无效或不可访问的绑定
- 验证方法：测试不存在的路径、普通文件、不可读目录、路径穿越，以及逃离允许根目录的符号链接。
- 预期结果：校验以安全且可操作的错误失败；任务不能启动；接口不会暴露无关目录内容。
- 关联需求点：无效绑定禁止执行；路径安全。

### CHK-005 现有增删改查兼容性
- 验证方法：增加 通用项目执行 字段前后，执行项目、任务、模板和报告的增删改查回归测试。
- 预期结果：现有接口与数据继续正常工作；新增或未知字段不会破坏 Markdown 往返读写。
- 关联需求点：兼容现有项目功能。

## 角色解析与执行准入

### CHK-006 项目默认角色
- 验证方法：为项目配置不同的默认执行 Agent 和 Reviewer，再创建不含角色覆盖的任务并检查执行尝试记录。
- 预期结果：执行尝试使用项目默认角色，并记录稳定的 Provider 与 Agent 身份。
- 关联需求点：项目默认执行者和 Reviewer。

### CHK-007 任务级角色覆盖
- 验证方法：为单个任务设置不同于项目默认值的执行 Agent 和 Reviewer。
- 预期结果：只有该任务使用覆盖值；其他任务仍使用项目默认角色。
- 关联需求点：任务可以覆盖默认角色。

### CHK-008 缺少 Reviewer 或执行者自审
- 验证方法：分别以没有可解析 Reviewer、Reviewer 不可用、执行者与 Reviewer 为同一 Agent 的配置启动任务。
- 预期结果：在调用 Provider 前拒绝启动并给出明确错误；执行 Agent 无法自审。
- 关联需求点：必须具有独立 Reviewer。

### CHK-009 明确选择执行任务
- 验证方法：创建多个待办任务，然后指定其中一个任务启动。
- 预期结果：只有指定任务启动；优先级和排序不会导致其他任务被选择；完成后不会自动执行下一项。
- 关联需求点：用户手动选择任务；不自动连续处理待办列表。

### CHK-010 单项目只允许一个活动任务
- 验证方法：一个任务执行期间，通过界面和 API 尝试启动同一项目的第二个任务。
- 预期结果：第二次启动返回包含活动任务信息的冲突；不会创建第二个 Provider 执行操作。
- 关联需求点：每个项目同一时间只执行一个任务。

### CHK-011 Git 脏工作区确认
- 验证方法：在有未提交修改的 Git 工作区启动任务，获取风险提示和状态指纹；确认一次后再次修改工作区，并复用旧确认重试。
- 预期结果：首次启动暂停并要求明确确认；匹配当前状态的确认只能启动一次；过期或重复确认被拒绝。
- 关联需求点：未提交修改必须由用户确认风险。

### CHK-012 干净 Git 与非 Git 目录准入
- 验证方法：分别在干净 Git 仓库和有效普通目录中启动任务。
- 预期结果：两者都不要求脏工作区确认，但仍必须通过路径和角色校验。
- 关联需求点：根据工作区类型执行正确的准入流程。

## Provider 无关的任务执行

### CHK-013 执行 Agent 的 Provider 矩阵
- 验证方法：使用可控适配器，分别由 OpenClaw、Hermes 和 Codex 执行等价任务。
- 预期结果：三种 Provider 生成相同的规范化执行状态和证据结构，同时保留各自 Provider 引用。
- 关联需求点：现有三类 Provider 均可执行项目任务。

### CHK-014 工作区传递
- 验证方法：检查各 Provider 收到的任务上下文，并在隔离测试目录内执行无害的文件读取和写入。
- 预期结果：各 Provider 按对应策略使用项目绑定工作区，不会退回无关的默认目录。
- 关联需求点：Provider 项目绑定。

### CHK-015 执行活动与证据收集
- 验证方法：执行一个会修改文件，并分别运行成功和失败测试的任务。
- 预期结果：执行记录包含执行者总结、修改文件、命令及结果、任务 checklist、时间信息和规范化活动引用。
- 关联需求点：执行过程可追踪；为最终验收提供证据。

### CHK-016 证据脱敏与容量限制
- 验证方法：生成包含模拟密钥、敏感环境变量和超大 diff 或命令输出的证据。
- 预期结果：持久化和展示内容均经过脱敏与容量限制；存在明确截断标记；模拟敏感值不会泄漏。
- 关联需求点：证据安全、隐私和可读性。

### CHK-017 执行失败
- 验证方法：让各 Provider 在产生部分活动后返回错误或超时。
- 预期结果：任务进入阻塞；保留部分证据和错误分类；不会开始 Reviewer 流程或进入完成状态。
- 关联需求点：失败后保留证据并等待用户处理。

### CHK-018 取消执行
- 验证方法：在任务已修改文件后，通过界面和 API 取消活动任务。
- 预期结果：只中断当前 Provider 操作；任务进入阻塞；已有修改和证据保留；界面不暗示已自动回滚。
- 关联需求点：取消语义与证据保留。

## 独立审查与最终验收：独立 Reviewer 与最终验收

独立审查与最终验收 必须建立在已验收的 执行底座 执行尝试和证据结构之上，补充独立 Reviewer、返工、阻塞升级和用户最终验收。

## 独立 Reviewer 与返工

### CHK-019 Reviewer 的 Provider 矩阵
- 验证方法：分别使用 OpenClaw、Hermes 和 Codex Reviewer 审查可控证据包，每次使用不同的执行 Agent。
- 预期结果：三种 Provider 都返回相同的规范化 Reviewer 结果结构，并记录稳定的 Reviewer 身份。
- 关联需求点：任意现有 Provider 的 Agent 均可承担 Reviewer。

### CHK-020 Reviewer 只读隔离
- 验证方法：构造要求或尝试修改文件的 Reviewer 响应，对比审查前后的工作区状态指纹。
- 预期结果：Reviewer 只收到 Office 生成的证据包，不会建立可写执行通道；工作区内容保持不变。
- 关联需求点：Reviewer 不得修改项目文件。

### CHK-021 Reviewer 结构化结果校验
- 验证方法：测试合法的 `pass`、`needs_more_work`、`blocked`，以及格式错误、字段不全、未知状态和提示注入响应。
- 预期结果：只有通过结构校验的结果可以推动状态变化；格式错误或不可信指令无法通过或完成任务。
- 关联需求点：Reviewer 结果结构化；状态流转安全。

### CHK-022 Reviewer 证据覆盖
- 验证方法：检查包含 checklist、修改文件、测试、执行总结和历史返工记录的审查证据包。
- 预期结果：必需证据均存在，或明确标记为缺失、不可用或已截断；Reviewer 理由能关联到 checklist 和证据。
- 关联需求点：Reviewer 预验收依据完整。

### CHK-023 Reviewer 通过后等待用户
- 验证方法：让 Reviewer 返回完整通过结果，检查任务、项目板、API 和持久化状态。
- 预期结果：任务进入“等待用户验收”，不会进入 Done；刷新后仍可查看验收证据。
- 关联需求点：用户承担最终验收。

### CHK-024 自动返工循环
- 验证方法：连续三次返回 `needs_more_work` 并检查每次派发。
- 预期结果：原执行 Agent 收到结构化问题和返工历史并进行修改，随后再次由独立 Reviewer 审查；每轮记录都被保留。
- 关联需求点：自动退回原执行者；最多三轮返工。

### CHK-025 返工次数上限
- 验证方法：完成三轮返工后，再返回一次未通过结果。
- 预期结果：不会开始第四轮自动返工；任务进入阻塞，并展示完整返工历史和阻塞原因。
- 关联需求点：三轮返工安全上限。

### CHK-026 Reviewer 阻塞或无法判断
- 验证方法：分别返回 `blocked`、证据不足和连续无法解析的 Reviewer 响应。
- 预期结果：任务进入阻塞，不会被误判通过或无限重试；用户可以看到原因和已有证据。
- 关联需求点：Reviewer 无法判断时升级给用户。

## 用户最终验收

### CHK-027 最终验收证据界面
- 验证方法：Reviewer 通过后，在桌面和移动端打开最终验收界面。
- 预期结果：Checklist 结果、执行者总结、修改文件、测试结果、Reviewer 结论、返工历史和风险提示清晰可读且不重叠；不要求用户阅读完整原始轨迹。
- 关联需求点：最终验收必须展示的内容及可用性。

### CHK-028 用户验收通过
- 验证方法：对“等待用户验收”的任务执行通过操作，然后重新加载项目并检查仓库。
- 预期结果：任务进入 Done，并记录操作者、时间和执行尝试；系统不会自动提交、推送、合并、部署或改写文件。
- 关联需求点：用户明确通过；不自动执行 Git 操作。

### CHK-029 用户拒绝并返工
- 验证方法：填写反馈并拒绝验收，确认重新执行，然后再次取得 Reviewer 通过。
- 预期结果：非空反馈被记录；旧 Reviewer 通过结果失效；默认退回原执行 Agent，且必须重新通过 Reviewer 后才能再次等待用户验收。
- 关联需求点：用户拒绝后重新走完整预验收流程。

### CHK-030 用户标记阻塞
- 验证方法：为等待验收的任务填写原因并标记阻塞。
- 预期结果：任务进入阻塞，保留验收证据包并记录用户原因，不会发起新的 Provider 调用。
- 关联需求点：用户可主动标记阻塞。

### CHK-031 Done 状态防绕过
- 验证方法：尝试通过拖拽、通用任务更新、批量排序、旧版 review-check 接口、自动模式和过期验收请求，将未验收任务移动到 Done。
- 预期结果：所有绕过方式均被拒绝或不改变规范状态；只有对当前执行尝试的有效用户验收才能完成任务。
- 关联需求点：禁止绕过最终用户验收。

## 恢复、可观测性与回归

### CHK-032 页面刷新恢复
- 验证方法：分别在执行、审查、返工、阻塞和等待用户验收状态刷新页面。
- 预期结果：界面恢复相同的活动任务、执行尝试、状态、证据和可用操作，不会重复派发任务。
- 关联需求点：状态持久化与过程可追踪。

### CHK-033 服务重启对账
- 验证方法：在 Provider 操作未结束时重启服务，并在等待用户验收时再次重启服务。
- 预期结果：无法证明仍在运行的操作保守转为阻塞并保留证据；等待验收状态原样恢复；任务不会静默完成或自动重新派发。
- 关联需求点：安全恢复。

### CHK-034 过期与重复操作保护
- 验证方法：对更新后的执行尝试重放旧的 Reviewer、取消、返工和验收请求，并并发发送重复请求。
- 预期结果：通过执行尝试 ID 和状态转换守卫，以幂等方式拒绝过期或重复修改。
- 关联需求点：状态完整性与并发安全。

### CHK-035 审计记录
- 验证方法：分别完成通过、返工、阻塞、取消和用户验收场景，检查持久化活动记录。
- 预期结果：每次状态变化均记录项目、任务、执行尝试、操作者、前后状态、时间和原因，且不包含凭据。
- 关联需求点：项目执行可审计。

### CHK-036 现有 Provider 与 Codex live bridge/6 回归
- 验证方法：执行 OpenClaw/Hermes 聊天测试、Codex provider/bridge/server/Codex live activity controls E2E 测试，以及项目增删改查和旧工作流兼容测试。
- 预期结果：现有聊天、Codex 实时活动、审批、补充输入、取消、恢复，以及非 通用项目执行 项目行为均保持可用。
- 关联需求点：不破坏现有 Provider 和 Codex 控制能力。

### CHK-037 人工端到端验收
- 验证方法：在真实本地工作区中，使用至少两种不同 Provider 的独立执行 Agent 和 Reviewer，人工完成一个直接通过任务和一个拒绝返工任务。
- 预期结果：浏览器中可以理解并控制完整生命周期；只有用户明确验收才能进入 Done；刷新后所有证据仍然存在。
- 关联需求点：通用项目执行 核心成功标准。

### CHK-038 执行底座 人工端到端验收
- 验证方法：在真实本地工作区中，分别使用至少两种 Provider 的执行 Agent，完成项目绑定、角色配置、脏工作区确认、指定任务启动、活动查看、刷新恢复和取消或失败处理。
- 预期结果：项目执行底座可独立使用；成功任务停在“执行完成”并保留证据；不会触发旧版执行者自审、自动执行下一任务或绕过验收进入 Done。
- 关联需求点：执行底座 独立交付与验收标准。

## 明确顺延的验收范围

- 通用项目执行 不验收同一项目内的并发任务。后续 Phase 必须覆盖隔离 worktree、分支归属、合并与冲突处理，以及并发证据隔离。
- 通用项目执行 不验收 Office 自有定时调度和通用自动化；这些能力需要单独的后续需求与 checklist。
