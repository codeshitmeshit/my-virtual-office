## 需求背景

`app/game.js` 是 Virtual Office 的 canvas 像素风办公室主渲染与交互文件，当前 16,568 行，是所有 JS 文件中最大的单体文件。它承载了过多职责：配置持久化、分支管理、家具编辑器、天气系统、光照/时间系统、相机/缩放/平移、碰撞检测、Agent 渲染（外观+发型+服装+头饰+眼镜+道具）、全量办公室环境绘制、主菜单/设置面板、技能工作坊、会议看板、技能库等。这导致：

- 代码导航困难，函数查找和定位成本高。
- 修改的风险半径极大，任何一个改动都可能影响无关模块。
- 无法对单一模块做独立测试。
- 多人协作和 code review 困难。

## 目标

将 `game.js` 拆分为多个职责清晰的 ES6 模块，每个模块独立加载，保持当前功能完整不变。

## 目标用户

开发者（自己及未来的维护者）。

## 范围

- **In scope**：将 `game.js` 按功能域拆分为独立 JS 文件，通过 `<script>` 标签或 ES6 `import`/`export` 加载。
- **In scope**：确保拆分后页面渲染、交互、所有 UI 面板功能与当前行为完全一致。
- **In scope**：消除当前 `var` 声明变为模块作用域的 `const`/`let`，避免全局命名污染。

## 非目标

- 不进行功能重构或行为变更。
- 不重写渲染管线或 Agent 类的内部逻辑。
- 不改动后端 `server.py`。
- 不引入构建工具（webpack/vite 等）——保持零构建依赖。

## 关键约束

- 当前项目通过 `<script src="...">` 标签在 HTML 中直接加载 JS 文件，所有函数和变量都在全局作用域。拆分后必须保持相同的加载方式（或使用 `type="module"`），不能引入构建步骤。
- `i18n.js`、`chat.js` 等其他文件通过全局变量和函数名与 `game.js` 交互，拆分不能破坏这些全局接口。
- 当前没有构建系统，所有文件直接由浏览器加载。

## 当前已知结论

- 代码审查已识别出 29 个逻辑模块，可以按以下分组拆分（详见 `review.md`）。
- 使用 IIFE 包裹或 ES6 模块均可，但 ES6 模块更干净且现代浏览器均支持。
- 拆分后需在 HTML 中更新脚本加载顺序。
