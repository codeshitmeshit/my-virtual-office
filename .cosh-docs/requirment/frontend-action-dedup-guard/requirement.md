# Frontend Action Dedup Guard

## 背景

项目侧前端存在大量 inline `onclick` 触发异步写操作的按钮。当前很多操作依赖刷新、轮询或延时展示来更新 UI 状态，但在网络延迟、后端处理慢、确认弹窗未关闭或列表尚未刷新时，用户可以重复点击同一个按钮，从而发出重复请求。

已观察到的典型入口包括项目执行启动、项目流水线启动、会议 blocker 继续/阻塞/重开、启动审查、验收/退回/标记阻塞、定时任务运行/启停/删除等。

## 目标用户

- 使用 Project Execution 的普通用户。
- 在项目任务、会议 blocker、验收、定时任务等关键流程中执行操作的项目负责人。
- 维护 Virtual Office 前端项目侧状态机的开发者。

## 目标

1. 为项目侧关键写操作增加统一的前端防重机制。
2. 用户点击后应立即看到按钮进入处理中状态，避免误以为没有响应而重复点击。
3. 同一业务动作在进行中时，重复点击应被忽略或给出轻量提示，不再重复发请求。
4. 确认弹窗和文本输入弹窗的提交按钮也必须防重。
5. 防重释放应覆盖成功、失败、取消、异常和刷新后状态变化。

## 范围

首批覆盖 `app/projects.js` 中高风险项目侧写操作：

- 单任务 Project Execution 启动。
- 项目级 Project Execution 启动与重启流水线。
- 停止当前执行或指定 attempt。
- 会议 blocker：继续执行、标记阻塞、重新申请会议。
- 启动审查。
- 验收通过、退回返工、标记阻塞。
- 工作流 start/stop。
- Scheduled cron：创建/编辑、立即运行、暂停/恢复、启用/禁用、删除。
- 弹窗确认按钮和文本输入提交按钮。

## 非目标

- 不改变后端状态机语义。
- 不以本需求替代后端幂等、状态校验或 attempt/version 检查。
- 不重构全部项目管理 UI。
- 不覆盖聊天侧、会议独立页面或主菜单设置页的所有按钮；除非它们直接复用本次项目侧弹窗提交路径。

## 关键约束

- 保持当前无构建步骤的前端结构，不引入新依赖。
- 尽量局部修改 `app/projects.js`，避免把现有 inline `onclick` 大规模迁移成事件委托。
- 防重 key 必须按业务动作粒度生成，不能用一个全局锁阻塞无关操作。
- 操作完成后必须释放锁；如果成功后会刷新项目状态，应在刷新结束后释放或依赖重渲染清掉旧按钮状态。
- 对失败路径要恢复按钮状态，避免按钮永久 disabled。
- 对话框确认按钮必须同步禁用，避免同一弹窗提交多次。

## 当前已知结论

- `app/projects.js` 的 API wrapper 是裸 `fetch`，没有请求级去重。
- 多数按钮通过 inline `onclick` 调用 async action，没有统一 `inFlight`、`pendingAction`、`busy` 或 disabled 保护。
- 个别保存描述操作有局部禁用按钮和延迟恢复，但不是通用机制。
- 后端部分接口有状态机兜底，但前端仍可能在刷新前发出重复请求，带来错误 toast、重复弹窗、重复状态切换和可观测性噪音。

## 产品澄清结论

- 重复点击反馈：只对高风险操作给出轻量 toast，普通重复点击可以静默忽略或仅保持 busy 状态，避免噪音。
- 处理中状态释放时机：请求返回并且页面状态刷新完成后再释放，覆盖刷新前旧按钮仍可点击的窗口期。
- 首期覆盖范围：覆盖项目执行关键链路和 scheduled cron，不做全量 projects.js 写操作兜底。
- 失败后重试规则：按错误类型区分；普通网络或后端错误可恢复重试，状态冲突类错误应刷新状态后再允许继续。
- 成功标准：同时看请求数、用户可感知的处理中反馈和状态一致性，验收以关键路径自动化为准。
