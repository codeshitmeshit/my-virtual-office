# 本期汉化完整性修复方案评审

需求名：`localization-completion`

## 评审结论

结论：通过，无阻塞问题，可以进入测试 checklist 确认阶段。

当前需求范围由既有汉化机制和上一轮代码审计界定，不需要新增产品交互或服务端协议。实现可在现有前端本地化边界内完成。

## 产品评审

### 范围是否明确

明确。目标是修复项目现有用户界面的中英文切换完整性，不扩展到用户内容、代理输出、开发文档和新语言。

### 用户预期

- 选择中文后，导航、配置、操作、反馈和错误提示均可理解。
- 品牌、模型、协议、命令和文件名可以保留英文。
- 语言切换不能改变业务数据或操作结果。

### 产品风险

- “完全没有英文”不适合作为机械标准，因为技术缩写、品牌名和用户数据必须保留。
- 服务端返回的原始错误可能为英文。界面应汉化错误上下文，但保留原始错误详情用于排障。
- 示例数据中的人名、模型名和文件内容不属于未汉化。

上述边界已写入需求，不构成阻塞。

## 技术评审

### 架构

保留现有 `i18n.js + locales/*.json + data-i18n*` 架构。建议补充统一的前端 `_t(key, params)` 访问方式，并让动态组件在 `i18n:changed` 后重绘或刷新其固定文案。

不建议本期引入第三方框架，避免扩大改动面和部署依赖。

### 静态 DOM

- 文本节点使用 `data-i18n`。
- `title` 使用 `data-i18n-title`。
- `placeholder` 使用 `data-i18n-placeholder`。
- 仅对语言包内受控 HTML 使用 `data-i18n-html`。
- 同一元素不要嵌套多个会互相覆盖 `textContent` 的翻译节点。

官网现有富文本词条必须改用 HTML 翻译入口，或拆分成纯文本子节点。

### 动态 DOM

动态组件不能只在创建时写死英文。建议采用以下顺序：

1. 将固定文案提取为翻译键。
2. 使用参数插值处理数量、名称和状态。
3. 业务数据经过现有转义函数后再拼接。
4. 监听 `i18n:changed`，重新渲染当前打开的动态组件。

涉及 `innerHTML` 的代码必须区分受控模板和外部数据，不能把服务端返回值直接拼入未转义 HTML。

### 翻译键管理

- 中英文语言包新增键必须成对提交。
- 修复 `save_failed` 等缺失键。
- 对 `proj_priority_` 等动态前缀键使用显式白名单或测试展开，避免静态扫描误报。
- 建议增加脚本检查：语言包键集合一致、代码引用键存在、JSON 可解析。

### 状态与数据

业务状态值继续使用既有英文标识，例如 `working`、`active`、`completed`；只翻译展示层。不得修改 API 参数、存储字段或条件判断。

### 错误处理

错误提示采用“本地化上下文 + 原始详情”：

```text
保存失败：<原始错误>
```

这样兼顾中文可理解性和排障信息完整性。

### 权限与安全

本需求不改变权限模型。主要安全风险是富文本翻译和动态 HTML 引入 XSS：

- 语言包 HTML 只能包含代码库内受控内容。
- 用户、代理和服务端数据必须转义。
- 不使用翻译字符串构造可执行属性或事件处理器。

### 兼容性

- 保持当前浏览器语言自动检测和 `localStorage` 偏好。
- 验证首次中文浏览器、已有英文偏好和运行中切换三种情况。
- 不依赖网络字体或外部翻译服务完成汉化。

### 性能

语言切换时允许重绘当前动态面板，但不应重新请求无关业务数据。语言包规模较小，不需要懒加载改造。

### 可观测性

现有 i18n 失败仅输出控制台警告。本期至少应保证：

- 加载失败不导致页面脚本崩溃。
- 缺失键可通过自动化扫描发现。
- 浏览器控制台不出现新增异常。

### 测试可行性

可通过以下层级验证：

- Node 脚本检查 JSON、键集合和引用键。
- JavaScript 语法检查。
- 现有测试回归。
- 浏览器分别以中文和英文打开主要页面并执行关键动态操作。
- 针对富文本、确认框、错误状态和语言切换做人工检查。

## 建议实施分组

1. 完善 i18n 基础设施和静态键检查。
2. 修复主页面静态 DOM。
3. 修复 `game.js` 动态编辑器、会议和技能功能。
4. 修复聊天、设置向导、模型和定时任务页。
5. 修复官网富文本与页面元信息。
6. 统一中文术语并执行双语回归。

## 方案变更规则

如果实施中决定引入第三方 i18n 框架、改变语言代码、改变 API 状态值，或扩大到翻译代理输出，则属于方案变更，必须重新评审并重新确认 checklist。
