# Auto Project Workspace Checklist

确认状态：已确认

## 验收规则

- 本 checklist 确认前不得生成 todolist。
- 所有验证都应在可控临时项目和临时工作区中执行。
- 自动创建的工作区必须被视为正式项目工作区。
- 普通项目能力必须保留。

## Checklist

### CHK-001 默认创建可执行项目
- 验证方法：打开创建项目表单，检查项目类型默认状态；或通过默认创建流程提交一个未填写工作区路径的项目。
- 预期结果：默认创建的是 Project Execution 项目，而不是普通项目。
- 关联需求点：默认项目类型为可执行项目。

### CHK-002 未填写工作区时自动创建工作区
- 验证方法：创建默认可执行项目，不填写工作区路径。
- 预期结果：项目创建成功；项目包含有效 `workspacePath`、`workspaceKind`、`workspaceStatus`，且 `projectExecutionEnabled` 为 true。
- 关联需求点：空工作区路径自动创建正式工作区。

### CHK-003 自动工作区命名
- 验证方法：创建标题明确的项目，例如 `Demo Project`，检查生成的工作区路径或目录名。
- 预期结果：工作区名称包含项目标题的安全化形式和时间戳，例如 `demo-project-20260616143536`。
- 关联需求点：自动工作区使用项目名加时间戳命名。

### CHK-004 自动工作区可见反馈
- 验证方法：创建未填写工作区路径的默认可执行项目后进入项目看板或项目详情。
- 预期结果：用户能看到生成的工作区路径和状态；不会只显示空路径或“未配置工作区”。
- 关联需求点：自动创建后明确展示路径和状态。

### CHK-005 手动工作区路径仍可用
- 验证方法：创建默认可执行项目并填写一个有效工作区路径。
- 预期结果：项目绑定用户填写的路径，不额外创建随机工作区；Project Execution 仍启用。
- 关联需求点：保留用户手动指定工作区能力。

### CHK-006 普通项目入口
- 验证方法：在创建项目表单中切换为普通项目并提交。
- 预期结果：项目创建成功；`projectExecutionEnabled` 为 false；没有自动创建工作区；Project Execution 专属面板和产物入口不默认出现。
- 关联需求点：仍允许创建普通项目。

### CHK-007 普通项目优先级低于可执行项目
- 验证方法：检查创建项目表单的信息架构和默认状态。
- 预期结果：可执行项目是默认选择；普通项目通过明确开关可选，但不会成为默认。
- 关联需求点：普通项目保留但默认是可执行项目。

### CHK-008 自动工作区创建失败
- 验证方法：构造自动工作区无法创建的环境或配置，再创建默认可执行项目且不填写路径。
- 预期结果：项目创建失败；用户看到明确失败原因；系统不会静默降级为普通项目。
- 关联需求点：自动创建失败时项目创建失败。

### CHK-009 API 层默认行为
- 验证方法：直接调用项目创建 API，提交 `projectExecutionEnabled: true` 且不提供 `workspacePath`。
- 预期结果：后端自动创建并绑定工作区；行为不依赖前端表单。
- 关联需求点：后端必须支持默认可执行项目的真实创建逻辑。

### CHK-010 API 创建普通项目
- 验证方法：直接调用项目创建 API，提交普通项目意图且不提供工作区。
- 预期结果：创建普通项目；不自动创建工作区；`projectExecutionEnabled` 为 false。
- 关联需求点：普通项目能力在 API 层保留。

### CHK-011 工作区来源记录
- 验证方法：分别创建自动工作区项目和手动工作区项目，检查项目数据。
- 预期结果：系统能区分自动创建工作区和用户提供工作区，以便删除流程正确判断。
- 关联需求点：删除自动工作区前需要询问，且不能误删用户工作区。

### CHK-012 删除自动工作区时询问
- 验证方法：删除一个使用自动工作区的项目。
- 预期结果：删除前询问是否一并删除自动工作区；用户可以选择保留或删除。
- 关联需求点：项目删除时询问是否删除自动工作区。

### CHK-013 删除时保留自动工作区
- 验证方法：删除自动工作区项目时选择保留工作区。
- 预期结果：项目被删除；自动工作区目录仍存在；用户能看到或理解保留结果。
- 关联需求点：删除项目不应强制删除正式工作区。

### CHK-014 删除时删除自动工作区
- 验证方法：删除自动工作区项目时选择一并删除工作区。
- 预期结果：项目被删除；对应自动工作区也被删除；不影响其他项目或用户目录。
- 关联需求点：用户确认后可清理自动工作区。

### CHK-015 用户提供工作区不被误删
- 验证方法：删除一个绑定用户手动路径的 Project Execution 项目。
- 预期结果：不会默认删除用户提供的工作区；不会把用户路径当作系统自动工作区处理。
- 关联需求点：数据安全和工作区来源边界。

### CHK-016 产物入口可用
- 验证方法：创建默认可执行项目且不填写工作区，进入项目后打开产物入口。
- 预期结果：产物入口可用；没有 Markdown 时显示空状态；有 Markdown 时可列出和打开。
- 关联需求点：自动工作区用于产物展示。

### CHK-017 Project Execution 启动可用
- 验证方法：创建默认可执行项目且不填写工作区，添加任务并配置执行与审查 Agent，然后启动任务。
- 预期结果：任务可以进入 Project Execution 流程；执行使用自动创建的工作区。
- 关联需求点：自动工作区用于任务执行和文件修改。

### CHK-018 现有项目兼容
- 验证方法：加载已有普通项目、已有手动工作区 Project Execution 项目和已有自动工作区项目数据。
- 预期结果：已有项目状态不被错误迁移；普通项目仍普通；已有可执行项目仍可执行。
- 关联需求点：兼容现有项目数据。

### CHK-019 模板创建一致性
- 验证方法：从模板创建项目，不填写工作区路径并保持默认可执行项目。
- 预期结果：模板项目同样自动创建工作区并启用 Project Execution；如果切换普通项目，则不自动创建。
- 关联需求点：创建项目入口行为一致。

### CHK-020 回归测试
- 验证方法：执行项目 CRUD、Project Execution 基础流程、artifact 管理相关自动化和前端语法检查。
- 预期结果：既有项目管理、执行审查验收、产物管理能力不因默认工作区变更而回归。
- 关联需求点：不破坏现有 Project Execution 和项目管理行为。
