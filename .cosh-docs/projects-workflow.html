Markdown 存储

MarkdownProjectStore（project_store.py:64）将项目/任务存储为 Markdown 文件：

{STATUS_DIR}/
  projects-md/
    {slug}--{project_id}/
      project.md          # 项目元数据 (frontmatter) + 描述 + 活动日志
      tasks/
        {slug}--{task_id}.md  # 任务元数据 + 描述 + 评论 + 附件

Markdown 格式示例：

---
id: abc123
title: Website Redesign
status: active
priority: high
columns_json: [{"id":"c1","title":"To Do"}]
---
# Project
Redesign the company website
## Activity
- [2026-06-01T10:00:00Z] (user) Created project
为什么用 Markdown 而非 JSON？

相比纯 JSON，Markdown + frontmatter 格式让人类可以直接编辑文件，同时保持了结构化数据的能力。复杂字段（columns、tags）序列化为 JSON 存储在 frontmatter 中。

工作流管道

_wf_run_pipeline() 函数实现自动任务分派：

1

读取配置

读取项目任务和列配置

2

找到待处理任务

找到 "To Do" 列中的任务

3

分派给代理

根据 assignee 分发给代理，通过 workflow session 发送任务

4

监听完成

监听代理完成状态，如果有 review 列，进入审查循环

5

自动模式

autoMode 下持续处理下一个任务

审查循环

工作流引入了复杂的 self-review 机制：

  • 最多 5 轮 self-review 迭代循环
  • Review 需要工具调用证据（read/exec/browser），纯文本 PASS 会被拒绝
  • 四阶段管道：Backlog → In Progress → Review → Done
历史 Bug

commit eaa3986 修复了 review 解析器的无限循环问题（sentiment fallback + cycle safety net）。详见server.py 的增长密码。

自动恢复机制

_wf_auto_resume_on_startup() 在服务器启动时恢复卡住的任务：

_wf_auto_resume_on_startup()
  |
  +-- sleep(3) 等待服务器就绪
  +-- 扫描 projects.json 中卡在 "In Progress"/"Review" 的任务
  +-- 检查是否有对应的 workflow session
  +-- 如果 session 状态为 done/running/failed，恢复管道
