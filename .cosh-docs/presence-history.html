Before：office.py 手动模式

在 Gateway Presence 引入之前，Agent 状态更新依赖 office.py CLI 工具手动操作：

  • 用户需要手动运行 CLI 命令来更新 Agent 状态
  • 状态更新不是实时的，需要手动触发
  • 无法自动感知 Agent 的实际活动（工具调用、消息处理等）
问题

手动模式无法反映 Agent 的真实工作状态，用户体验差，且容易因为忘记更新状态而产生误导。

设计亮点

设计选择实现原因
只读观察者不向 Gateway 发送任何命令不干扰 Agent 正常运行，即使 Presence 崩溃 Agent 仍工作
asyncio.Queue 单读者事件通过 Queue 传递给单一消费者保证并发安全，避免多线程竞争
sessions.list 轮询每 10 秒轮询一次补充实时事件，确保不遗漏状态变化
手动 override TTL30 秒自动过期允许短暂手动设置，但不会永久覆盖自动推导
磁盘快照每 30 秒保存状态崩溃恢复，防止内存状态丢失

后续演进

从 cd65ffd 到 2026-06-05，gateway_presence.py 经历了 16 次修改：

  • 状态机细化：从简单的 working/idle 演变为 working/finishing/idle 三态，增加 grace period
  • Run/Tool 精确追踪：引入 runId 和 toolCallId 追踪，避免误判
  • 超时保护：IDLE_TIMEOUT_SEC = 120s，FINISHING_GRACE_SEC = 12s，ACTIVE_RUN_STALE_SEC = 6h
  • 断线重连：指数退避（5s → 60s），确保 Gateway 断开后自动恢复
结论

Gateway-first Presence 架构是项目最核心的设计决策之一，由 Elix 在初始发布当天亲自引入。它将状态更新从"手动推"转变为"自动拉"，大幅提升了用户体验和系统可靠性。
