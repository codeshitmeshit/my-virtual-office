Provider-neutral Agent 调用链路

这条链路解释为什么 Project Execution 和 A2A 通信可以同时支持 OpenClaw、Hermes、Codex、Claude Code。关键不是某一个 provider，而是 server 层根据 `providerKind` 分派，再把返回值归一化成统一 result。

详细链路说明

1

`_handle_agent_platform_comm_send` 校验

输入 fromAgentId、toAgentId、message、conversationId；动作是校验发送方、目标方和消息；输出 from_ref、to_ref。

2

写入办公室可见 request event

`_append_comm_event` 写入 visibleInOffice request，让跨平台通信不是私下 CLI 调用。

3

按 providerKind 分发

Codex/Claude Code 走专用 chat handler，其他路径进入 `_wf_call_agent`，输出 provider_result 或 reply。

4

`_wf_call_agent` fallback

它先识别 Hermes/Codex，再尝试 OpenClaw HTTP、WebSocket、CLI fallback，输出 reply string。

5

`normalize_provider_result`

把 ok、status、reply、threadId、sessionId、tools、tokenUsage、modifiedFiles、needsHumanIntervention 归一化。

6

adapter 内部执行

`CodexProvider`、`HermesProvider`、`ClaudeCodeProvider` 把 profile/message/session 转成各自 runtime 请求。

7

写 reply event

最终回复和 provider metadata 被写入通信历史，并返回给 UI 或 attempt 状态机。

统一 provider 返回合同

normalized = {
    "ok": ok,
    "status": status,
    "reply": result.get("reply") or "",
    "providerKind": provider_kind,
    "threadId": thread_id or result.get("threadId") or "",

不同 provider 的 thread、session、run、tools、文件变化被映射到统一字段。

分支、失败路径和数据变化

分支human source、agent source、archive guard、codex、claude-code、Hermes、OpenClaw HTTP/WS/CLI fallback。
失败路径缺 fromAgentId 返回 400；目标不存在 404；provider exception 变成 ok false；HTTP status 由 provider_http_status 映射。
数据流caller body -> office refs -> comm event -> provider envelope -> adapter result -> normalized metadata -> reply event。

关键概念补充

1. providerKind

出现位置：roster、provider handler、execution result。它决定同一个 Agent 调用要进入 Codex、Hermes、Claude Code 还是 OpenClaw 路径。

2. visibleInOffice

出现位置：agent platform communication event。它让跨平台通信出现在办公室历史中，而不是不可见的后台调用。
