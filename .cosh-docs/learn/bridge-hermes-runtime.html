Hermes API / CLI 双路径 bridge

Hermes bridge 同时保留 CLI adapter 和 native API client。服务器聊天路径会优先尝试 native API run/SSE/approval；不可用或未启用时回落 `hermes chat -Q -q`。

入口：POST /api/hermes/runs终止：ProviderRunBridge 或 Hermes terminal result置信度：中
详细链路代码摘录数据流验证概念

详细链路说明

1. run handler 创建 VO run

`_handle_hermes_run_start` 创建 runId、events queue、idempotency scope，并注册 ProviderRunBridge。

2. worker 注入 `_onHermesApiEvent`

worker 给 `_handle_hermes_chat` 增加 native event callback，用于把 Hermes API event 映射成 SSE。

3. chat handler 判断 API 能力

Hermes 配置中的 `apiEnabled`、API health/capabilities 决定是否走 `HermesApiClient`。不满足则使用 CLI fallback。

4. API 路径提交 run

`HermesApiClient.start_run` POST `/v1/runs`，并可携带 session key、profile、message。随后 `stream_run_events` 读取 `/v1/runs/{runId}/events`。

5. native event 映射

server 把 message/tool/reasoning/approval 等 Hermes event 映射成 ProviderRunBridge 事件，同时写 progress history。

6. CLI fallback 路径

`HermesProvider.send_chat_message` 运行 `hermes chat -Q [--resume session] -q message`，从 stdout/stderr 提取 session_id 和 reply。

7. terminal 与 stop

成功时 emit `run.completed`；失败时 `run.failed`。stop 只有 native API 路径能调用 `/v1/runs/{runId}/stop`，CLI fallback stop 不对等。

这段是 CLI fallback 的真实边界。

# app/providers/hermes.py
def start_run(...): POST /v1/runs
def stream_run_events(...): GET /v1/runs/{run_id}/events
def respond_approval(...): POST /v1/runs/{run_id}/approval
def stop_run(...): POST /v1/runs/{run_id}/stop

API 路径提供 run、SSE、approval、stop，能力比 CLI fallback 更接近后台 run 模型。

关键概念补充

Capability-gated API path：Hermes 不是只看配置开关，还要看 native server 是否声明 run submission 和 SSE event 能力。

Fallback asymmetry：CLI fallback 能完成 chat，但 stop、approval、实时 event 能力不完全等价。
