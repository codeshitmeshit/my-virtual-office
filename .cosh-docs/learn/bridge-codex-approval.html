Codex approval / interaction loop

Codex app-server 会反向向 VO 发起 server request，要求命令、文件、权限或用户输入。VO 的默认策略是 fail closed：没有可交互用户时取消并中断 turn；允许交互时才把请求挂起并等待 `/api/codex/approval/respond`。

入口：item/*/requestApproval终止：respond 或 needs_human_intervention置信度：高
详细链路代码摘录数据流验证概念

详细链路说明

1. Codex 发出 server request

request method 属于 `APPROVAL_METHODS`，例如 `item/commandExecution/requestApproval` 或 `item/tool/requestUserInput`。

2. runtime reader 分到 `_handle_server_request`

JSONL message 同时有 `id` 和 `method`，被 `JsonlAppServerRuntime` 识别为 server request。

3. 找到当前 operation

Codex request 带 threadId，client 用 threadId 查 `_operations`。找不到 operation 时无法关联 UI pending。

4. 允许交互则挂起

如果 operation.allow_interaction 为 true，request 写入 `pending_requests`，approval 写入 `_pending_approvals`，并 emit `interaction` event。

5. 不允许交互则 fail closed

VO 自动返回 cancel/decline/empty answers，并发送 `turn/interrupt`，operation result 变成 `needs_human_intervention`。

6. 用户响应 approval

`/api/codex/approval/respond` 进入 `CodexProvider.respond_approval`，再调用 bridge 的 respond 方法，把 accept/cancel 回写给 app-server。

7. 原 turn 继续或终止

app-server 收到批准后继续发 agentMessage/turn completed；取消时 turn interrupted，VO 输出错误或人工介入状态。

关键概念补充

Fail-closed approval boundary：没有明确用户响应时，VO 不替用户批准 provider 的命令或文件操作，而是取消并中断，降低自动化误操作风险。

Interaction event：Codex 的 server request 被转成 VO 可观察的 pending interaction，用于 UI 或 API 响应。
