前端项目执行启动链路

这条链路解决的问题是：用户在项目看板点击“启动项目”或“启动此任务”时，前端如何把 UI 状态转成后端能理解的 project execution 请求，并处理 dirty workspace、缺 reviewer、无可执行任务等确认分支。

详细链路代码节点代码摘录分支和失败概念

详细链路说明

1

`projectExecutionProjectStartAction`

输入是当前 project、启动模式、dirtyFingerprint 和用户确认状态；动作是组装 `mode`、`restartPipeline`、`skipReviewConfirmed`；输出交给 `api.projectExecutionProjectStart`。证据：`app/projects.js:3443`。

2

`api.projectExecutionProjectStart`

把 projectId、mode、dirtyFingerprint 通过 POST 发到 `/api/projects/{projectId}/project-execution/start`，输出 JSON response。证据：`app/projects.js:330`。

3

`OfficeHandler.do_POST`

输入是 HTTP path；动作是匹配项目级 start 路由；输出调用 `_handle_project_execution_project_start`。证据：`app/server.py:22489`。

4

`_handle_project_execution_project_start`

检查 project execution 是否启用、是否已有 active task、是否 restart，并选择 `_project_execution_next_task`。输出是 selected task 或 error。证据：`app/server.py:15809`。

5

`_handle_project_execution_start`

项目级启动最终复用任务级 handler。输入是 selected task 和 `projectStart=true`，动作是校验 workspace、roles、dirty worktree、startMode，并创建 attempt。证据：`app/server.py:15736`。

6

前端处理 response

如果返回 confirmationRequired，前端显示确认对话并重入；如果 started，设置 workflow active 并启动 polling；如果 error，展示错误。证据：`app/projects.js:3450`。

关键代码摘录

前端项目级启动 API

app/projects.jsapi.projectExecutionProjectStart
fetch(`/api/projects/${projectId}/project-execution/start`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ mode, dirtyFingerprint, skipReviewConfirmed, restartPipeline })
})

这段代码证明前端不执行任务，只把启动意图和确认状态交给后端。

项目级启动复用任务级启动

app/server.py_handle_project_execution_project_start
task = _project_execution_next_task(project)
result = _handle_project_execution_start(
    project_id,
    task.get("id"),
    {**body, "projectStart": True},
)

项目级启动只是选出下一条任务，真正的 attempt 创建和执行仍由任务级 handler 完成。

分支、失败路径和数据变化

关键分支缺 reviewer 需要确认、dirty worktree 需要 fingerprint、没有可执行任务、已有 active task、restartPipeline 要求任务可重复。
失败路径executor 缺失返回 409；workspace 无效返回 409；dirty 未确认返回 confirmationRequired。
数据流UI mode -> POST body -> selectedTask -> attempt record -> workflow state -> polling status。

关键概念补充

1. dirtyFingerprint

出现位置：`app/projects.js` 与 `_handle_project_execution_start`。它是前端确认当前工作区 dirty 状态的凭据，避免用户确认后状态又变化。

2. workflowActive

出现位置：`app/server.py` project/task 状态。它让前端知道项目执行总线正在运行，需要进入 polling。
