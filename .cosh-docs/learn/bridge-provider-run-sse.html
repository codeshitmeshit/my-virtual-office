ProviderRunBridge：统一后台 run 和 SSE

这条链路解释为什么 Codex、Claude Code、Hermes 的 `/runs` 可以共享一套 SSE 输出。它不是 provider 执行器，只保存 run meta、事件队列和 terminal result。

入口：ProviderRunBridge.remember(meta)终止：stream_events terminal event置信度：高
解决什么详细链路代码摘录数据与分支验证

这条链路解决什么问题

原始 provider 调用通常是阻塞的，且事件格式各不相同。ProviderRunBridge 把“后台运行中的一组事件”抽象成 runId + queue。run handler 负责执行 provider，bridge 只负责存取 meta、写入 queue、按 SSE 协议输出。

详细链路说明

1. run handler 构造 meta

输入是 provider run 请求和 agent 信息，输出是 `runId`、`profile`、`conversationId`、`events = queue.Queue()`。

2. `remember(meta)` 注册

`ProviderRunBridge` 在锁内把 `runId` 映射到 meta。缺少 runId 的 meta 会被忽略。

3. worker 调用 `emit`

worker 用 `emit(run_id, event_name, payload)` 把 provider progress 变成 queue item。`emit` 会自动补 runId、agentId、profile。

4. SSE 请求进入 `stream_events`

如果 runId 不存在，直接输出 404 和 `run.failed`。如果存在，响应头设置为 `text/event-stream`。

5. 循环消费 queue

有事件时写 `event: name` 与 JSON `data`。没有事件但未完成时每 10 秒写 keepalive。

6. terminal fallback

当 meta.done 且 queue 为空，bridge 根据 result.ok/status 输出 completed/cancelled/failed，然后 break。

7. 清理

finally 中如果 meta.done，则 `clear(run_id)`。各 worker 也安排 600 秒 timer 兜底清理。

关键代码摘录

# app/server.py: ProviderRunBridge 核心字段
def remember(self, meta):
    if not isinstance(meta, dict) or not meta.get("runId"):
        return
    with self._lock:
        self._runs[str(meta["runId"])] = meta

def emit(self, run_id, event_name, payload=None):
    meta = self.get(run_id)
    events = meta.get("events")
    events.put_nowait({"event": event_name, "data": payload, "ts": int(time.time() * 1000)})

这段证明 bridge 只维护内存 registry 和 queue，不启动 provider。
