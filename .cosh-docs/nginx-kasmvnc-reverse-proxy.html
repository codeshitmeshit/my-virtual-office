Nginx 与 KasmVNC 反向代理

适用于 Virtual Office 运行在本机 7243/7244，KasmVNC 运行在本机 6901，并通过同一 HTTPS 域名的 /browser/ 子路径访问的部署方式。

端口与路径

公网路径上游用途
/http://127.0.0.1:7243Virtual Office HTTP/API/静态资源
/wshttp://127.0.0.1:7244Virtual Office WebSocket
/browser/https://127.0.0.1:6901/KasmVNC 页面、资源和 Viewer WebSocket

推荐 Nginx 配置

以下配置由当前可用配置脱敏整理。替换域名和证书路径即可使用，Kasm 凭据通过单独的受限文件引入。

server {
    listen 443 ssl;
    server_name office.example.com;

    ssl_certificate     /etc/nginx/certs/office.example.com/fullchain.pem;
    ssl_certificate_key /etc/nginx/certs/office.example.com/private.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_session_timeout 5m;

    location / {
        proxy_pass http://127.0.0.1:7243;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
    }

    location = /ws {
        proxy_pass http://127.0.0.1:7244;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
    }

    location = /browser {
        return 301 /browser/;
    }

    location /browser/ {
        proxy_pass https://127.0.0.1:6901/;
        proxy_ssl_verify off;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host 127.0.0.1:6901;
        include /etc/nginx/snippets/kasm-upstream-auth.conf;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
        proxy_buffering off;
    }
}

server {
    listen 80;
    server_name office.example.com;
    return 301 https://$host$request_uri;
}
为什么可以按当前配置复制

HTTP、Office WebSocket、KasmVNC 静态资源与 wss://office.example.com/browser/websockify 已分别验证。复制时只应替换域名、证书和认证信息。

Kasm 上游认证片段

不要把真实用户名密码或 Base64 值写进仓库中的 Nginx 示例。生成认证值后放到仅 root 可读的文件。

# 生成值，输出内容不要提交到 Git
printf '%s' '<KASM_USERNAME>:<KASM_PASSWORD>' | base64 -w0

# /etc/nginx/snippets/kasm-upstream-auth.conf
proxy_set_header Authorization "Basic <BASE64_KASM_CREDENTIALS>";

sudo chown root:root /etc/nginx/snippets/kasm-upstream-auth.conf
sudo chmod 600 /etc/nginx/snippets/kasm-upstream-auth.conf
不要复制当前生产凭据

Base64 只是编码，不是加密。文档、日志、截图和代码仓库中都不应出现真实认证值。

Viewer URL 与 WebSocket 路径

环境变量可以只保存 Viewer 页面地址，Virtual Office 前端会自动补充正确的 Kasm WebSocket 子路径。

VO_VIEWER_URL=https://office.example.com/browser/

# 实际 iframe URL 会包含：
https://office.example.com/browser/?path=browser/websockify&resize=scale&autoconnect=1

# 最终 WebSocket：
wss://office.example.com/browser/websockify
本次 lastActiveAt 报错的原因

缺少 path=browser/websockify 时，KasmVNC 会尝试连接站点根路径 /websockify。连接失败后，KasmVNC 4.0 的空闲定时器继续访问已释放的 RFB 对象，于是显示 lastActiveAt 异常。

配置与启动验收

sudo nginx -t
sudo nginx -s reload

curl -fsS https://office.example.com/health

# 项目启动后检查：
cat data/startup-health.json

启动报告至少应包含 http: true、websocket: true、browser.cdpAvailable: true 和 viewerWebsocket.ok: true。

关键注意事项

  • 保留尾部斜杠：location /browser/ 与 proxy_pass https://127.0.0.1:6901/; 的尾部斜杠决定 URI 重写结果。
  • 必须转发 Upgrade：Kasm 页面和 WebSocket 共用 /browser/ location，不能只按普通静态页面代理。
  • 不要只检查 CDP：CDP 9224 可用不代表 Kasm Viewer WebSocket 可用，必须检查 viewerWebsocket.ok。
  • 避免缓存旧 iframe：修改 Viewer URL 构造逻辑后，浏览器需强制刷新一次。
  • 限制 6901 暴露范围：推荐仅绑定回环地址或通过防火墙限制，不直接公开 KasmVNC 上游端口。
